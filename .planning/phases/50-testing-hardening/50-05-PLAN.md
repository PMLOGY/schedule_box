Output: 3 partition DDL scripts, 1 rollback script, 1 batch migration TypeScript script.

  <action>
    Create 3 raw SQL DDL migration files and 1 rollback script. Follow the conservative pattern from research.

    **0004_partition_bookings.sql (HARD-01):**
    1. `CREATE TABLE bookings_partitioned (LIKE bookings INCLUDING DEFAULTS INCLUDING CONSTRAINTS INCLUDING INDEXES) PARTITION BY RANGE (start_time);`
    2. Create monthly partitions from 2025-01-01 through 2027-06-01 (safety buffer back + 15 months forward). Name pattern: `bookings_YYYY_MM`. Use UTC timestamps for range boundaries.
    3. Create `bookings_default` DEFAULT partition for out-of-range data.
    4. Per research: btree_gist exclusion constraints CANNOT be on the partitioned parent in PG 16. The application already uses SELECT FOR UPDATE as the primary double-booking guard. Add a comment documenting this explicitly. Do NOT attempt to add EXCLUDE USING gist on the parent.
    5. Recreate all indexes from the original table on the partitioned parent (they will propagate to partitions automatically). The key indexes: `(company_id, start_time)` composite, `(uuid)` unique, `(employee_id, start_time)`, `(status)`.
    6. Do NOT include the data migration or RENAME swap in this DDL file — that is handled by the batch migration script.

    **0005_partition_notifications.sql (HARD-02):**
    Same pattern but partitioned by `scheduled_at` (or `created_at` if scheduled_at is nullable — check schema). Monthly partitions 2025-01 through 2027-06. Default partition.

    **0006_partition_audit_logs.sql (HARD-02):**
    Same pattern partitioned by `created_at`. Monthly partitions 2025-01 through 2027-06. Default partition. audit_logs is in the `platform` schema — check if it uses a schema prefix or public schema.

    **rollback_partitions.sql:**
    For each table, the rollback is:
    ```sql
    BEGIN;
    ALTER TABLE bookings RENAME TO bookings_partitioned;
    ALTER TABLE bookings_old RENAME TO bookings;
    COMMIT;
    ```
    Repeat for notifications and audit_logs. Include a header comment: "Run this ONLY if the original _old tables still exist. After verification period, drop _old tables."

    Important: Keep Drizzle schema files (bookings.ts, platform.ts, notifications.ts) UNCHANGED. Drizzle defines the logical shape; partitioning is applied via raw SQL only. Drizzle introspection cannot express PARTITION BY RANGE.

<task type="auto">
  <name>Task 2: Batch migration script with RENAME swap</name>
  <files>packages/database/scripts/partition-migrate.ts</files>
  <action>
    Create a TypeScript script that:
    1. Connects to PostgreSQL using the DATABASE_URL env var (use postgres.js, already in devDependencies)
    2. For each table (bookings, notifications, audit_logs):
       a. Apply the corresponding DDL file (0004/0005/0006) to create the partitioned shadow table
       b. Batch-migrate data: `INSERT INTO {table}_partitioned SELECT * FROM {table} ORDER BY id LIMIT 500 OFFSET :n` in a loop until no more rows
       c. Log progress: "Migrated {count}/{total} rows for {table}"
       d. After all data migrated, do the RENAME swap in a single transaction:
          ```sql
          BEGIN;
          ALTER TABLE {table} RENAME TO {table}_old;
          ALTER TABLE {table}_partitioned RENAME TO {table};
          COMMIT;
          ```
    3. After all 3 tables are swapped, run EXPLAIN on a sample query to verify partition pruning:
       ```sql
       EXPLAIN (FORMAT TEXT) SELECT * FROM bookings WHERE company_id = 1 AND start_time >= '2026-03-01' AND start_time < '2026-04-01';
       ```
       Log the explain output. If it shows "Seq Scan on bookings_2026_03" (partition name), pruning is working.

    4. Accept a `--dry-run` flag that runs DDL + batch migration but skips the RENAME swap.
    5. Accept a `--table` flag to migrate only one table (for incremental rollout).
    6. Accept a `--rollback` flag that runs the rollback SQL.

    Script runs via: `npx tsx packages/database/scripts/partition-migrate.ts`
    With dry run: `npx tsx packages/database/scripts/partition-migrate.ts --dry-run`
    With rollback: `npx tsx packages/database/scripts/partition-migrate.ts --rollback`

    Important: This script should be run manually during a maintenance window, NOT as part of CI/CD. Add a prominent warning comment at the top.

    Per research: Do NOT batch-migrate and RENAME in the same transaction. The batch migration can be interrupted and resumed. The RENAME swap must be fast (just catalog changes).
