<task type="auto">
  <name>Task 1: Wire booking emails into create and status transition routes</name>
  <files>apps/web/app/api/v1/bookings/route.ts, apps/web/lib/booking/booking-transitions.ts</files>
  <action>
1. Read the booking creation route (`apps/web/app/api/v1/bookings/route.ts`) to find where a booking is successfully inserted. After the successful insert (after the DB transaction commits), add:

   a) Insert a notification record into the `notifications` table with channel='email', status='pending', recipient=customer email, subject='Potvrzeni rezervace', body=plain text version.

   b) Fire-and-forget call to `sendBookingConfirmationEmail()` from `lib/email/booking-emails.ts`. On success, update notification status to 'sent' and set sentAt. On failure, update to 'failed' with errorMessage.

   c) Also insert a second notification record for the SMS reminder with channel='sms', status='pending', scheduledAt = booking.startTime minus 24 hours. This row will be picked up by the cron job.

   Pattern for fire-and-forget:
   ```typescript
   sendBookingConfirmationEmail({ to, customerName, serviceName, ... })
     .then(async () => {
       await db.update(notifications).set({ status: 'sent', sentAt: new Date() }).where(eq(notifications.id, notifRecord.id));
     })
     .catch(async (err) => {
       console.error('[BookingEmails] Confirmation failed:', err);
       await db.update(notifications).set({ status: 'failed', errorMessage: String(err) }).where(eq(notifications.id, notifRecord.id));
     });
   ```

2. Read the booking transition logic. Look for files that handle confirm/cancel/complete transitions — likely in `apps/web/app/api/v1/bookings/[id]/route.ts` or a shared `booking-transitions.ts` or inline in PATCH/PUT handlers. Find the actual file by grepping for `confirmBooking\|cancelBooking\|completeBooking\|status.*confirmed\|status.*cancelled\|status.*completed` in the bookings API directory.

3. After each status transition (confirmed, cancelled, completed), add:
   - Insert notification record (channel='email', status='pending')
   - Fire-and-forget `sendBookingStatusChangeEmail({ ...data, newStatus })` with same .then/.catch status update pattern
   - Use Czech subjects: confirmed="Rezervace potvrzena", cancelled="Rezervace zrusena", completed="Rezervace dokoncena"

4. CRITICAL: Email failure must NEVER roll back the booking. All email calls must be outside the DB transaction and wrapped in .catch().

5. If SMTP env vars are not set (dev environment), the email functions should gracefully skip (check in booking-emails.ts or guard here).

<task type="auto">
  <name>Task 2: Create SMS reminder cron route + vercel.json cron config</name>
  <files>apps/web/app/api/v1/cron/sms-reminders/route.ts, vercel.json</files>
  <action>
1. Create `apps/web/app/api/v1/cron/sms-reminders/route.ts`:

   - Export `GET` handler (Vercel Cron invokes via GET).
   - Security: Validate `Authorization: Bearer ${CRON_SECRET}` header. Return 401 if missing/wrong. CRON_SECRET is already in env.ts.
   - Query notifications table: `channel = 'sms' AND status = 'pending' AND scheduledAt <= NOW() AND scheduledAt >= NOW() - INTERVAL '30 minutes'`. The 30-minute window ensures we catch reminders that might have been missed in the previous cycle.
   - For each pending SMS notification:
     a) Call `sendSMS(notification.recipient, notification.body)` from `lib/sms/twilio-client.ts`.
     b) On success: update notification status to 'sent', set sentAt = new Date().
     c) On failure: update status to 'failed', set errorMessage.
   - Process in batch — up to 50 per cron invocation to stay within Vercel function timeout.
   - Return JSON response: `{ processed: N, sent: N, failed: N }`.
   - Use `export const runtime = 'nodejs'` — Twilio SDK requires Node.js runtime (not Edge).
   - Use `export const maxDuration = 30` — allow enough time for batch processing.

2. Create or update `vercel.json` in project root:
   - If vercel.json exists, read it first and add crons array.
   - If it doesn't exist, create it with:
   ```json
   {
     "crons": [
       {
         "path": "/api/v1/cron/sms-reminders",
         "schedule": "*/15 * * * *"
       }
     ]
   }
   ```
   - The 15-minute schedule means reminders are sent within a 15-minute window of the 24h-before mark.

3. Ensure the GET /api/v1/notifications route (existing) properly returns notification records with status field — this satisfies NOTIF-04 (delivery status visible to owner). Read the existing route to confirm it already returns status; if not, add status to the select fields.
