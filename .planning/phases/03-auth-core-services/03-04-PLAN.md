  <files>

  <action>
    Create `apps/web/app/api/v1/auth/register/route.ts`:
    - Export POST handler (no auth required — public endpoint)
    - Validate body with registerSchema (from validations/auth.ts)
    - Use db.transaction to:
      1. Create company (name from company_name, generate slug from name using slugify, email from user email)
      2. Hash password with hashPassword()
      3. Find 'owner' role ID from roles table
      4. Create user with companyId, roleId, email, passwordHash, name, phone
      5. Insert initial password into password_history
    - Generate JWT pair with generateTokenPair()
    - Return 201 with { access_token, refresh_token, expires_in, user: { uuid, email, name, role: 'owner', company_id } }
    - Handle duplicate email: catch unique constraint violation, return 409 ConflictError
    - Slug generation: lowercase name, replace spaces/special chars with hyphens, append random 4-char suffix for uniqueness

    Create `apps/web/app/api/v1/auth/login/route.ts`:
    - Export POST handler (no auth required — public endpoint)
    - Validate body with loginSchema
    - Find user by email (SELECT with JOIN on roles for role name)
    - If no user, return 401 UnauthorizedError('Invalid email or password') — generic message, don't reveal which is wrong
    - If user not active (is_active=false), return 401
    - Verify password with verifyPassword()
    - If invalid, return 401 (TODO: increment failed_login_count, lock after 5 failures — placeholder comment)
    - If user.mfaEnabled and no mfa_code in request:
      - Generate mfa_token using nanoid(32)
      - Store in Redis: `mfa:{mfa_token}` = userId, TTL 300 seconds (5 min)
      - Return 200 { mfa_required: true, mfa_token }
    - If user.mfaEnabled and mfa_code provided:
      - Import { authenticator } from 'otplib' (installed by this plan)
      - Verify TOTP code inline: authenticator.verify({ token: mfa_code, secret: user.mfaSecret })
      - If invalid, return 401 UnauthorizedError('Invalid MFA code')
      - NOTE: Plan 03-05 will create lib/auth/mfa.ts with reusable verifyMFACode(). Once available, this inline check can be refactored to use verifyMFACode() for consistency. For now, inline otplib usage is correct and self-contained.
    - Generate JWT pair
    - Update user.lastLoginAt
    - Return 200 { access_token, refresh_token, expires_in, user: { uuid, email, name, role, company_id, mfa_enabled, email_verified } }
    - Set refresh_token as httpOnly cookie: { httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'strict', maxAge: 30*24*60*60, path: '/api/v1/auth/refresh' }

    Create `apps/web/app/api/v1/auth/refresh/route.ts`:
    - Export POST handler (no auth required — uses refresh token, not access token)
    - Accept refresh_token from request body OR from httpOnly cookie
    - Call rotateRefreshToken(refreshToken) from lib/auth/jwt.ts
    - Return 200 with new { access_token, refresh_token, expires_in }
    - Set new refresh_token as httpOnly cookie (same settings as login)

    Install otplib: `cd apps/web && pnpm add otplib`
  </action>

    Check that refresh reads from both body and cookie.
  </verify>

<task type="auto">
  <name>Task 2: Create logout, password management, email verification, and profile endpoints</name>
  <files>
    apps/web/app/api/v1/auth/logout/route.ts
    apps/web/app/api/v1/auth/forgot-password/route.ts
    apps/web/app/api/v1/auth/reset-password/route.ts

  </files>
  <action>
    Create `apps/web/app/api/v1/auth/logout/route.ts`:
    - Export POST handler (requires auth)
    - Extract access token from Authorization header
    - Call blacklistToken(token) to add to Redis blacklist
    - Revoke all refresh tokens for user: UPDATE refresh_tokens SET revoked=true WHERE userId=user.sub
    - Actually: find user by UUID (user.sub) to get internal userId, then revoke
    - Clear refresh_token cookie
    - Return 204 noContentResponse()

    Create `apps/web/app/api/v1/auth/forgot-password/route.ts`:
    - Export POST handler (public, no auth)
    - Validate body with forgotPasswordSchema
    - Find user by email
    - If user exists:
      - Generate reset token using nanoid(64)
      - Hash token with SHA-256
      - Store in Redis: `password_reset:{hash}` = userId, TTL 3600 (1 hour — per research recommendation)
      - Log reset link to console: `console.log('[DEV] Password reset link: /reset-password?token=${resetToken}')`
      - In production, this would send email via notification service (not yet built)
    - ALWAYS return 200 { message: 'If an account with that email exists, a password reset link has been sent' } — don't reveal whether email exists

    Create `apps/web/app/api/v1/auth/reset-password/route.ts`:
    - Export POST handler (public, no auth)
    - Validate body with resetPasswordSchema
    - Hash provided token with SHA-256
    - Look up in Redis: `password_reset:{hash}`
    - If not found, throw BadRequestError('Invalid or expired reset token')
    - Check password history with checkPasswordHistory(userId, newPassword)
    - If password was used before, return 400 ValidationError('Password was recently used')
    - Call updatePassword(userId, newPassword)
    - Delete Redis key
    - Disable MFA on user (mfaEnabled=false, mfaSecret=null) — per research pitfall #3
    - Revoke all refresh tokens for user
    - Return 200 { message: 'Password has been reset successfully' }

    Create `apps/web/app/api/v1/auth/verify-email/route.ts`:
    - Export POST handler (public, no auth)
    - Validate body with verifyEmailSchema
    - Hash token, look up in Redis: `email_verify:{hash}`
    - If not found, throw BadRequestError('Invalid or expired verification token')
    - Update user: emailVerified=true
    - Delete Redis key
    - Return 200 { message: 'Email verified successfully' }

    Create `apps/web/app/api/v1/auth/change-password/route.ts`:
    - Export POST handler (requires auth)
    - Validate body with changePasswordSchema
    - Find user by UUID (from JWT sub)
    - Verify current_password against user.passwordHash
    - If invalid, throw UnauthorizedError('Current password is incorrect')
    - Check password history
    - If reused, throw ValidationError('Password was recently used')
    - Call updatePassword()
    - Revoke all refresh tokens (force re-login)
    - Blacklist current access token
    - Return 200 { message: 'Password changed successfully' }

    Create `apps/web/app/api/v1/auth/me/route.ts`:
    - Export GET handler (requires auth)
      - Find user by UUID with role name (JOIN roles)
      - Return { data: { uuid, email, name, phone, avatar_url, role, company_id (from company UUID, not internal ID), mfa_enabled, email_verified, created_at } }
      - IMPORTANT: Return company UUID not company internal SERIAL ID in response
    - Export PUT handler (requires auth)
      - Validate body with userUpdateSchema
      - Update user fields (name, phone, avatar_url) WHERE uuid = user.sub
      - Return updated user profile (same shape as GET)

    For all endpoints that need to find a user by UUID: query users WHERE uuid = user.sub (JWT sub claim is user UUID).
  </action>

    Check that me endpoint returns UUID not SERIAL for company_id.
  </verify>
  <done>
