  <name>Task 1: Create auth E2E tests (registration and login)</name>
  <files>apps/web/e2e/tests/auth.spec.ts</files>
  <action>
Create `apps/web/e2e/tests/auth.spec.ts` with the following test scenarios:

Import LoginPage and RegisterPage from page-objects, test data from helpers/test-data.

**Test 1: "new user can register successfully"**
- Use a fresh browser context (no storageState — this tests unauthenticated flow)
- Use `test.use({ storageState: { cookies: [], origins: [] } })` to override the default authenticated state
- Navigate to `/register` via RegisterPage POM
- Fill all 5 fields: name, email (use unique email with timestamp), companyName, password (must meet complexity: 12+ chars, upper+lower+digit+special), confirmPassword
- Click submit
- Wait for success message: `expect(page.locator('text=Registration successful')).toBeVisible()` or check for green success div
- The form shows success then redirects to login after 2 seconds — optionally `page.waitForURL('**/login')`

**Test 2: "user can login with valid credentials"**
- Use fresh context (no storageState)
- Navigate to `/login` via LoginPage POM
- Fill email `test@example.com`, password `password123`
- Click submit
- Wait for redirect away from login: `page.waitForURL` matching dashboard
- Verify dashboard content is visible (e.g., sidebar, main content area)
- This test validates the login flow works end-to-end against the seeded test database

**Test 3: "login shows error with invalid credentials"**
- Use fresh context
- Navigate to `/login`
- Fill email `wrong@example.com`, password `wrongpassword`
- Click submit
- Verify error message appears: the login form shows `.text-destructive` div with error text
- Verify we remain on the login page (URL still contains 'login')

**Test 4: "login form validates email format"**
- Use fresh context
- Navigate to `/login`
- Fill email `not-an-email`, password `password123`
- Click submit (or trigger blur)
- Verify form validation message appears (Zod validation: "Invalid email address")
- User should NOT be redirected

**Important implementation notes:**
- The app uses next-intl with `localePrefix: 'as-needed'` and `defaultLocale: 'cs'`. For Czech locale, URLs may or may not have `/cs/` prefix. Use URL patterns like `**/login` not `/login` for waitForURL.
- Login form uses `useTranslations('auth.login')` — button text is locale-dependent. Use regex: `/prihlasit|sign in|submit/i`
- Register form uses `useTranslations('auth.register')` — similarly use i18n-safe patterns
- The register form submits to `POST /api/v1/auth/register` — this needs the real server running (webServer config handles this)
- For registration test, generate a unique email per run to avoid "already exists" errors: `e2e-${Date.now()}@example.com`
- Wrap registration and login-with-invalid-creds tests in `test.describe` with `test.use({ storageState: { cookies: [], origins: [] } })` so they run unauthenticated
  </action>

Create `apps/web/e2e/tests/booking.spec.ts` with the following test scenario:

Import BookingWizardPage from page-objects, mock helpers from helpers/mock-api, test data from helpers/test-data.

**Test 1: "user can create a booking end-to-end"**
- This test uses the default storageState (authenticated as test owner) — no `test.use` override needed
- Before navigating, set up API mocks using the mock helpers:
  - `mockServicesAPI(page, [MOCK_SERVICE])` — return a service list with at least one service
  - `mockEmployeesAPI(page, [MOCK_EMPLOYEE])` — return available employees
  - `mockAvailabilityAPI(page, [...slots])` — return available time slots for the selected date
  - Mock the booking creation endpoint: `page.route('**/api/v1/bookings', ...)` to return 201 with a booking object
- Navigate to `/bookings/new` via BookingWizardPage POM
- **Step 1 — Select Service:** Wait for services to load, click on service by name ('Strih damsky' from mock data), click Next
- **Step 2 — Select Date/Time:** Interact with date picker to select a date, select a time slot, click Next. If the date picker is a calendar component (react-day-picker), click a future date. For time slots, click the first available slot.
- **Step 3 — Customer Info:** Fill customer name, email, phone. These fields are in the Step3CustomerInfo component. Click Next.
- **Step 4 — Confirmation:** Review the booking summary. Click Confirm button.
- Verify success: After confirmation, the app should either redirect to a success page, show a success toast (sonner), or redirect to the bookings list. Check for success indicator.

**Test 2: "booking wizard shows error for missing fields"**
- Authenticated context
- Mock services API
- Navigate to booking wizard
- Try to proceed from Step 1 without selecting a service
- Verify error message or disabled Next button prevents proceeding

**Important implementation notes:**
- The booking wizard uses Zustand store (`useBookingWizard`) — state is local, not persisted
- Step 1 loads services from `/api/v1/services` via React Query — mock this to control data
- Step 2 loads availability — mock this to ensure slots are available
- The wizard POSTs to `/api/v1/bookings` on confirm — mock this to avoid needing real DB state beyond seed data
- Use `page.route()` for API mocking since these are client-side fetch calls from React Query
- If API mocking proves complex for the wizard flow (e.g., wizard checks data consistency), fall back to using the seeded database directly (db:setup provides services, employees). In that case, skip the service/employee mocks and test against real seed data. Document which approach was used.
- Date picker interaction: `react-day-picker` renders a calendar. Click a future date cell. The calendar may use `role="gridcell"` for day buttons.
- Time slot interaction: Depends on component implementation. Look for time-slot buttons or a dropdown. Use `page.getByText()` with time pattern like `/10:00|14:00/`.
