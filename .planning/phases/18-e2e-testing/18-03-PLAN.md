**Create `apps/web/e2e/tests/payment.spec.ts`:**

Import mock helpers from helpers/mock-api, test data from helpers/test-data.

Test 1: "payment flow completes with Comgate test mode"
- Uses authenticated storageState (default)
- Strategy: Mock the entire payment creation API response at the browser level using `page.route()`. The Comgate client makes server-side calls that `page.route()` cannot intercept. Instead, mock the Next.js API route response that the client-side code calls.
- Steps:
  1. Mock `**/api/v1/services*` to return test services
  2. Mock `**/api/v1/bookings` POST to return a booking with `payment_required: true` and a `booking_uuid`
  3. Mock `**/api/v1/payments/comgate/create` POST to return `{ data: { transactionId: 'TX-E2E-001', redirectUrl: 'https://payments.comgate.cz/test/TX-E2E-001' } }`
  4. Intercept the browser redirect to Comgate: `page.route('**/payments.comgate.cz/**', ...)` — fulfill with a redirect (302) back to the app's payment callback URL with success status, OR abort and simulate the webhook callback via `page.request.post()` to `/api/v1/payments/comgate/callback`
  5. Navigate through booking wizard (can use a simplified flow or navigate directly to a payment page if one exists)
  6. Verify the booking is marked as paid: check for payment confirmation message or status indicator
- Alternative approach if the app doesn't have a direct payment page: Navigate to bookings list, find a booking with pending payment, click "Pay Now", and then intercept the flow.
- If the payment flow involves a full redirect to an external page (Comgate), intercept with `page.route()` and simulate the return.

Test 2: "payment creation handles Comgate errors gracefully"
- Mock `**/api/v1/payments/comgate/create` to return 500 error
- Attempt to pay for a booking
- Verify error message is shown to user (not a crash/white screen)
- The app should display a user-friendly error via toast (sonner) or inline error

**Create `apps/web/e2e/tests/ai-fallback.spec.ts`:**

Test 1: "AI predictions return fallback when service is unavailable"
- Uses authenticated storageState
- The webServer runs with `NODE_ENV: 'test'` and no `AI_SERVICE_URL` configured, so AI service calls will fail naturally
- Navigate to the AI dashboard: `/ai`
- Click on a prediction feature (e.g., capacity forecast, no-show predictions)
- The circuit breaker (Opossum) should catch the failures and return fallback values (defined in `apps/web/lib/ai/fallback.ts`)
- Verify the page renders without errors — no crash, no unhandled error state
- Verify fallback data is displayed (look for "fallback" indicator if the UI shows it, or just verify data appears)
- Verify the page does NOT show an error/crash screen: `expect(page.locator('body')).not.toContainText(/unhandled|unexpected|500/i)`

Test 2: "AI health endpoint reports circuit breaker state"
- Make a direct API request to `/api/v1/ai/health` using `page.request.get()`
- Parse the JSON response
- Verify the response has `data.status` field with value in `['healthy', 'degraded', 'unhealthy']`
- Verify `data.state` is one of `['CLOSED', 'HALF_OPEN', 'OPEN']`
- This validates the circuit breaker health reporting works end-to-end

**Important implementation notes:**
- Payment flow: The Comgate client uses `COMGATE_API_URL` env var (defaults to `https://payments.comgate.cz`). In test env, we mock at the API route response level using `page.route()` on `/api/v1/payments/comgate/create` rather than trying to intercept server-side fetch calls.
- AI fallback: The circuit breaker config has `timeout: 5000ms`, `errorThresholdPercentage: 50`, `volumeThreshold: 5`. With no AI service URL, calls fail immediately. After 5 failures, circuit opens and fallback is used automatically.
- The AI dashboard is at `/ai` and has sub-sections for different predictions. Pick the most straightforward one to test.
- If the AI page requires specific data to render, mock the relevant API endpoints with `page.route()`.

<task type="auto">
  <name>Task 2: Add E2E test job to CI pipeline</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Read the current `.github/workflows/ci.yml` and add a new `e2e` job after the existing jobs.

Add this new job to the CI pipeline:

```yaml
  # =========================================
  # Job 5: E2E Tests (Playwright)
  # =========================================
  e2e:
    name: E2E Tests
    runs-on: ubuntu-latest
    needs: [lint, test, build]
    if: github.ref == 'refs/heads/main' || github.event_name == 'pull_request'
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: schedulebox
          POSTGRES_PASSWORD: schedulebox
          POSTGRES_DB: schedulebox
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Setup database
        run: pnpm --filter @schedulebox/database db:setup
        env:
          DATABASE_URL: postgresql://schedulebox:schedulebox@localhost:5432/schedulebox

      - name: Build Next.js
        run: pnpm --filter @schedulebox/web build
        env:
          DATABASE_URL: postgresql://schedulebox:schedulebox@localhost:5432/schedulebox
          REDIS_URL: redis://localhost:6379
          JWT_ACCESS_SECRET: test-access-secret-ci
          JWT_REFRESH_SECRET: test-refresh-secret-ci
          NEXT_PUBLIC_APP_URL: http://localhost:3000

      - name: Install Playwright browsers
        run: npx playwright install --with-deps chromium firefox webkit

      - name: Run E2E tests
        run: npx playwright test --config apps/web/e2e/playwright.config.ts
        env:
          DATABASE_URL: postgresql://schedulebox:schedulebox@localhost:5432/schedulebox
          REDIS_URL: redis://localhost:6379
          JWT_ACCESS_SECRET: test-access-secret-ci
          JWT_REFRESH_SECRET: test-refresh-secret-ci
          NEXT_PUBLIC_APP_URL: http://localhost:3000
          NODE_ENV: test

      - name: Upload Playwright report
        uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-report
          path: apps/web/e2e/playwright-report/
          retention-days: 14

      - name: Upload test results
        uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-test-results
          path: apps/web/e2e/test-results/
          retention-days: 7
```

Key decisions:
- `needs: [lint, test, build]` — E2E runs after lint, unit tests, and build all pass. This ensures we don't waste CI time on E2E if basic checks fail. Note: `build` job only runs on `main` branch, so for PRs, E2E depends only on `[lint, test]`. Adjust the `needs` to `[lint, test]` if build is not always required, since the e2e job does its own `pnpm build`.
  - IMPORTANT: Check the `if` condition on the `build` job. If build only runs on main (`if: github.ref == 'refs/heads/main'`), then e2e CANNOT depend on build for PRs. In that case, set `needs: [lint, test]` instead. The e2e job does its own build step anyway.
- Service containers: PostgreSQL 16 and Redis 7 match production stack. No RabbitMQ needed (event publishing is try/caught and gracefully handles missing broker).
- `db:setup` seeds the database with test data including test users.
- Build step creates the Next.js production build. The `webServer` in playwright.config.ts uses `pnpm start` which serves the pre-built app.
- `workers: 1` in CI (set via playwright.config.ts `workers: process.env.CI ? 1 : undefined`) prevents resource exhaustion.
- Playwright report is uploaded as artifact regardless of success/failure (`!cancelled()`).
- `NODE_ENV: test` ensures Comgate client uses test mode.
