       ```sql
         COUNT(*) FILTER (WHERE status IN ('sent', 'delivered')) as sent,

         COUNT(*) as total
       FROM notifications
       WHERE channel = 'sms'
         AND created_at > date_trunc('month', NOW())
       ```
     - Read `SMS_COST_PER_SEGMENT_CZK` from env (default 1.50) and `SMS_MONTHLY_COST_LIMIT_CZK` (default 5000).
     - Exact segment count is tracked in the worker process (not available in the web app). Use `sent * 1.5` as rough estimate (average 1.5 segments per Czech UCS-2 SMS). This is acceptable for the monitoring API -- the worker-side monitoring scheduler in Plan 01 has precise segment tracking via prom-client counter.
     - Calculate estimatedCostCzk = sent * 1.5 * costPerSegment. Calculate percentOfLimit = (estimatedCostCzk / monthlyLimit) * 100.
     - Return JSON: `{ sent, failed, total, estimatedCostCzk, monthlyLimitCzk, percentOfLimit, month: new Date().toISOString().slice(0, 7) }`.

5. Create `apps/web/app/api/v1/monitoring/webhook-stats/route.ts`:
   - Same auth pattern as email-stats.
   - Export async function `GET(req: NextRequest)`:
     - Parse `?window=60` query param (default 60 minutes). Clamp to 1-1440 range.
     - Query processed_webhooks table:
       ```sql
       SELECT
         COUNT(*) FILTER (WHERE status = 'completed') as completed,
         COUNT(*) FILTER (WHERE status = 'failed') as failed,

         COUNT(*) as total
       FROM processed_webhooks
       WHERE processed_at > NOW() - make_interval(mins => ${windowMinutes})
       ```
     - Return JSON: `{ completed, failed, stuck, total, windowMinutes, checkedAt: new Date().toISOString() }`.

1. Modify `vitest.shared.ts`:
   - Add `'json-summary'` to the coverage.reporter array. Current value is `['text', 'json', 'html', 'lcov']`. Change to `['text', 'json', 'json-summary', 'html', 'lcov']`.
   - The `json-summary` reporter generates `coverage/coverage-summary.json` per package (a compact file with totals for lines/branches/functions/statements). This is what the CI step will parse. The existing `json` reporter produces `coverage-final.json` (per-file detail) which is not suitable for summary parsing.

2. Modify `.github/workflows/ci.yml` -- in the `test` job (Job 2: Unit Tests with Coverage Enforcement):
   - After the existing "Run unit tests with coverage" step and BEFORE the "Upload coverage report" step, add a new step:
     ```yaml
     - name: Generate coverage summary
       if: always()
       run: |
         echo "## Test Coverage Summary" >> $GITHUB_STEP_SUMMARY
         echo "" >> $GITHUB_STEP_SUMMARY
         echo "| Package | Lines | Branches | Functions | Statements |" >> $GITHUB_STEP_SUMMARY
         echo "|---------|-------|----------|-----------|------------|" >> $GITHUB_STEP_SUMMARY
         for pkg in packages/shared packages/events apps/web; do
           if [ -f "$pkg/coverage/coverage-summary.json" ]; then
             name=$(basename $pkg)
             lines=$(jq -r '.total.lines.pct // "N/A"' "$pkg/coverage/coverage-summary.json")
             branches=$(jq -r '.total.branches.pct // "N/A"' "$pkg/coverage/coverage-summary.json")
             functions=$(jq -r '.total.functions.pct // "N/A"' "$pkg/coverage/coverage-summary.json")
             statements=$(jq -r '.total.statements.pct // "N/A"' "$pkg/coverage/coverage-summary.json")
             echo "| **$name** | ${lines}% | ${branches}% | ${functions}% | ${statements}% |" >> $GITHUB_STEP_SUMMARY
           fi
         done
         echo "" >> $GITHUB_STEP_SUMMARY
         echo "_Threshold: 80% (enforced by vitest.shared.ts)_" >> $GITHUB_STEP_SUMMARY
     ```
   - The `if: always()` ensures the summary is generated even when coverage fails the threshold (so developers see WHICH package failed and by how much).
   - jq is pre-installed on ubuntu-latest GitHub Actions runners.
  </action>
