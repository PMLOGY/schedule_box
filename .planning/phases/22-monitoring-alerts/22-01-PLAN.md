<objective>
Instrument the notification worker with delivery metrics and a monitoring scheduler that detects email bounce rates above 5% and SMS costs approaching the monthly threshold, then sends alerts via email with optional Slack webhook fallback.

Purpose: MON-01 (email delivery monitoring) and MON-02 (SMS cost alerting) require the notification worker — the process that actually sends emails and SMS — to track delivery outcomes and periodically check for anomalies. Since Railway has no Prometheus/Alertmanager, alerting must be application-level via a BullMQ repeatable job.

Output: Worker-side monitoring metrics, instrumented email/SMS jobs, periodic monitoring scheduler, alert sender with dual-channel delivery (email + Slack).
</objective>

1. Create `services/notification-worker/src/monitoring/alert-sender.ts`:
   - Import nodemailer (already installed).
   - Import config from `../config.js`.
   - Define interface `AlertPayload { subject: string; body: string; severity: 'warning' | 'critical' }`.
   - Create a lazy SMTP transporter (only created on first alert send, using config.smtp settings — same pattern as auth-emails.ts module-level transporter).
   - Export async function `sendAlert(payload: AlertPayload): Promise<void>`:
     - Primary channel: send email via SMTP transporter to `config.monitoring.alertEmail` with subject prefixed by `[ScheduleBox ${payload.severity.toUpperCase()}]`. Wrap in try/catch — log failure but do NOT throw.
     - Fallback channel: if `config.monitoring.slackWebhookUrl` is non-empty, POST to the Slack webhook URL with `{ text: "${payload.subject}\n${payload.body}" }` using native `fetch()`. Wrap in try/catch — log failure but do NOT throw.
     - Always log the alert to console: `console.warn('[Monitor Alert]', payload.subject, payload.body)`.
   - The function must never throw — alerting failures should never crash the monitoring loop.

2. Create `services/notification-worker/src/monitoring/monitoring-scheduler.ts`:
   - Import { Queue, Worker } from 'bullmq'.
   - Import { db, notifications } from '@schedulebox/database'.
   - Import { sql } from 'drizzle-orm'.
   - Import { sendAlert } from './alert-sender.js'.
   - Import { config } from '../config.js'.
   - Import { smsSegmentsTotal, smsEstimatedMonthlyCostCzk } from './metrics.js'.
   - Define queue name constant: `MONITORING_QUEUE = 'monitoring-checks'`.
   - Export async function `startMonitoringScheduler(redisConnection: { host: string; port: number; password?: string }): Promise<{ queue: Queue; worker: Worker }>`:
     a. Create monitoring Queue with the redis connection.
     b. Add repeatable job: `queue.add('check-delivery-stats', {}, { repeat: { every: config.monitoring.checkIntervalMs }, jobId: 'delivery-stats-check' })`.
     c. Create Worker for the queue with handler `handleMonitoringCheck`.
     d. Return { queue, worker } for graceful shutdown.
   - Define async function `handleMonitoringCheck()`:
     a. **Email bounce rate check (MON-01):**
        - Query notifications table: `SELECT COUNT(*) FILTER (WHERE status IN ('sent', 'delivered', 'opened', 'clicked')) as delivered, COUNT(*) FILTER (WHERE status = 'failed') as failed, COUNT(*) as total FROM notifications WHERE channel = 'email' AND created_at > NOW() - INTERVAL '1 hour'`.
        - Use `db.execute(sql`...`)` with raw SQL (Drizzle ORM pattern).
        - Calculate failure rate: `failed / (delivered + failed)` (avoid division by zero — if total is 0, skip).
        - If failure rate > config.monitoring.emailBounceThreshold, call `sendAlert({ subject: 'Email bounce rate above threshold', body: 'Current rate: X.X% (threshold: Y%). Failed: N, Delivered: M in last hour.', severity: failureRate > 0.15 ? 'critical' : 'warning' })`.
     b. **SMS cost check (MON-02):**
        - Query notifications table for SMS: `SELECT COUNT(*) as total FROM notifications WHERE channel = 'sms' AND created_at > date_trunc('month', NOW())`.
        - Get current segment count from `smsSegmentsTotal` via `.get()` (prom-client Counter).
        - Estimate monthly cost: segments * config.monitoring.smsCostPerSegmentCzk.
        - Update the gauge: `smsEstimatedMonthlyCostCzk.set(estimatedCost)`.
        - If estimated cost > config.monitoring.smsMonthlyCostLimitCzk * 0.8, alert: `sendAlert({ subject: 'SMS cost approaching monthly limit', body: 'Estimated: X CZK, Limit: Y CZK, Messages this month: Z', severity: estimatedCost > smsMonthlyCostLimitCzk ? 'critical' : 'warning' })`.
     c. Log check completion: `console.log('[Monitor] Health check completed', { emailFailureRate, smsEstimatedCost })`.

3. Modify `services/notification-worker/src/queues.ts`:
   - No changes needed. The monitoring queue is created in monitoring-scheduler.ts with its own Queue instance (separate from notification queues). This avoids coupling.
