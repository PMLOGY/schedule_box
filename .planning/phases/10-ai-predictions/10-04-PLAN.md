  <files>

  <action>
    Create 4 Next.js API route handlers following the existing `createRouteHandler` factory pattern used throughout the project.

    **apps/web/app/api/v1/ai/predictions/no-show/route.ts:**
    - Use `createRouteHandler` with `permissions: ['bookings.read']` (owners and employees who can view bookings should see no-show predictions).
    - POST handler: Parse request body expecting `{ booking_id: number, features?: object }`.
    - Validate booking_id is a positive integer (use simple Zod schema inline or from shared schemas).
    - Call `predictNoShow.fire({ booking_id, features })` from `apps/web/lib/ai/client.ts`.
    - The circuit breaker `.fire()` method returns the prediction (or fallback if circuit is open/error).
    - Return `successResponse(prediction)` using existing response utility.
    - On error (catch), return error response with 503 status and a fallback prediction value using getNoShowFallback from fallback.ts.

    **apps/web/app/api/v1/ai/predictions/clv/route.ts:**
    - Use `createRouteHandler` with `permissions: ['customers.read']`.
    - POST handler: Parse `{ customer_id: number, features?: object }`.
    - Call `predictCLV.fire({ customer_id, features })`.
    - Return successResponse with prediction.
    - On error, return fallback with 503.

    **apps/web/app/api/v1/ai/predictions/health-score/route.ts:**
    - Use `createRouteHandler` with `permissions: ['customers.read']`.
    - POST handler: Parse `{ customer_id: number, recency_days: number, frequency: number, monetary: number }`.
    - Validate all fields with Zod (recency_days >= 0, frequency >= 0, monetary >= 0).
    - Call `predictHealthScore.fire({ customer_id, recency_days, frequency, monetary })`.
    - Return successResponse with prediction.
    - On error, return fallback.

    **apps/web/app/api/v1/ai/health/route.ts:**
    - Use `createRouteHandler` with `permissions: ['settings.read']` (admin-level to see AI system health).
    - GET handler: Call `getAIServiceStatus()` from client.ts.
    - Return successResponse with health status object including circuit breaker state and stats.

    IMPORTANT: Follow existing patterns exactly:
    - Import createRouteHandler from the existing middleware location (check apps/web/lib/middleware/ for exact path).
    - Import successResponse, errorResponse from existing response utility location.
    - Use the existing auth + RBAC pattern that all other API routes use.
    - The circuit breaker `.fire()` is how you invoke the breaker (opossum API).

  <name>Task 2: Add AI service to Docker Compose and update environment variables</name>
  <files>
    docker/docker-compose.yml
    .env.example
  </files>
  <action>
    **docker/docker-compose.yml:** Add the AI service container to Docker Compose:

    Add a new service `ai-service` after the `rabbitmq` service and before the `app` service:
    ```yaml
    ai-service:
      build:
        context: ../services/ai
        dockerfile: Dockerfile
      container_name: schedulebox-ai
      ports:
        - '8000:8000'
      environment:
        - ENVIRONMENT=development
        - AI_SERVICE_PORT=8000
        - REDIS_URL=redis://redis:6379
        - SCHEDULEBOX_API_URL=http://app:3000
        - MODEL_DIR=/app/models
        - ALLOWED_ORIGINS=http://localhost:3000,http://app:3000
        - LOG_LEVEL=debug
      depends_on:
        redis:
          condition: service_healthy
      healthcheck:
        test: ['CMD', 'python', '-c', 'import httpx; r = httpx.get("http://localhost:8000/health"); assert r.status_code in (200, 503)']
        interval: 10s
        timeout: 5s
        retries: 5
        start_period: 30s
      volumes:
        - ../services/ai:/app
        - ai_models:/app/models
      restart: unless-stopped
    ```

    Also update the `app` service to add the AI_SERVICE_URL environment variable:
    ```yaml
    # Add to app service environment:
    - AI_SERVICE_URL=http://ai-service:8000
    ```

    Add the `app` service dependency on `ai-service` (non-blocking - app should start even if AI is down, since we have circuit breaker fallback):
    Do NOT add ai-service as a hard dependency of app. The circuit breaker handles AI unavailability gracefully.

    Add the new volume to the volumes section:
    ```yaml
    ai_models:
    ```

    **IMPORTANT about healthcheck:** The Python healthcheck uses httpx which may not be available if the container hasn't started. Use a simpler alternative:
    ```yaml
    healthcheck:
      test: ['CMD-SHELL', 'python -c "import urllib.request; urllib.request.urlopen(\"http://localhost:8000/health\")" || exit 1']
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    ```
    Or even simpler using curl if available in the slim image. Since python:3.12-slim has urllib built-in, use that.

    **.env.example:** Add AI service environment variables at the end:
    ```
    # ===================
    # AI Service
    # ===================
    AI_SERVICE_URL=http://localhost:8000
    AI_SERVICE_PORT=8000
    AI_MODEL_DIR=./services/ai/models
    ```
