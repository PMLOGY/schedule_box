Output: Working auth flow where browser refresh preserves session, background refresh renews tokens every 12 minutes, and role routing is correct for all 4 user types.
</objective>

  <files>apps/web/stores/auth.store.ts, apps/web/hooks/use-auth.ts</files>
  <action>
**auth.store.ts changes:**

1. Add a `_hasHydrated` boolean field (default `false`) and a `setHasHydrated` action to the store.

2. In the `persist` config, add `onRehydrateStorage` callback that calls `setHasHydrated(true)` when rehydration completes:
```typescript
onRehydrateStorage: () => (state) => {
  state?.setHasHydrated(true);
},
```

3. Add a `startBackgroundRefresh` action that sets up a `setInterval` every 12 minutes (720000ms) to call `refreshToken()`. Store the interval ID in a module-level variable (not in state). Add a `stopBackgroundRefresh` action that clears it. The interval should only be active when `isAuthenticated` is true.

4. In the `login` action, after setting state, call `startBackgroundRefresh()`.

5. In the `logout` action, call `stopBackgroundRefresh()` before clearing state.

6. In `onRehydrateStorage`, if the rehydrated state has `isAuthenticated === true`, call `startBackgroundRefresh()` to resume refresh after page reload.

**use-auth.ts changes:**

7. Import `_hasHydrated` from the auth store (or subscribe to it).

8. Change the `useEffect` guard: if `!hasHydrated`, keep `isLoading = true` and return early (do NOT redirect). Only check `isAuthenticated` after hydration is complete. This prevents the flash-redirect-to-login on page reload.

9. The role-based routing logic in `useAuth` already handles role-path mismatch correctly (lines 38-74) -- keep it. But also add: if user is authenticated and on a public route (login/register), redirect them to `getHomeForRole(user.role)` so they don't see the login page while logged in.

**Cookie path mismatch fix (login/route.ts + refresh/route.ts):**

1. In `login/route.ts` line 212: Change cookie path from `'/api/v1/auth/refresh'` to `'/api/v1/auth'`. The cookie must be sent to ALL auth endpoints including `/api/v1/auth/refresh`. Currently the path is too narrow.

2. In `refresh/route.ts` line 59: The path is already `'/api/v1/auth'` -- keep it. Both endpoints must use the same path `'/api/v1/auth'`.

**Admin token refresh fix (jwt.ts):**

3. In `rotateRefreshToken` function, line 206: Change the check from:
```typescript
if (!user || !user.companyId) {
```
to:
```typescript
if (!user) {
```
Admin users (role='admin') and customer users (role='customer') have `companyId: null`. The function already passes `companyId ?? 0` to `generateTokenPair` (which handles 0 correctly). The null check was preventing these users from refreshing tokens entirely.

**API client refresh guard fix (api-client.ts):**

4. In the `request` method, lines 53-55: Remove the `isAdminUser` check that skips refresh for admin users. This was a workaround for the jwt.ts bug above. Now that admin refresh works, admin users should get the same auto-refresh behavior. Change:
```typescript
if (!isAdminUser) {
  // refresh logic
}
```
to just execute the refresh logic unconditionally (removing the `isAdminUser` guard and the `currentUser` lookup).

5. In the `catch` block (line 89-91), also reset `this.isRefreshing = false` and `this.refreshPromise = null` to prevent the refresh lock from being permanently stuck if a refresh fails.
