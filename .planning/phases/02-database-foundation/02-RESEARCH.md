# Phase 2: Database Foundation - Research

**Researched:** 2026-02-10
**Domain:** Database schema design, ORM configuration, PostgreSQL multi-tenancy
**Confidence:** HIGH

## Summary

Phase 2 requires implementing 47 database tables using Drizzle ORM with TypeScript, establishing Row Level Security for multi-tenant isolation, and implementing double-booking prevention mechanisms. The research reveals that Drizzle ORM is well-suited for this use case with its TypeScript-first approach, automatic type inference, and support for PostgreSQL-specific features including RLS, constraints, and partitioning.

The documentation provides a complete SQL schema (lines 896-2225) that must be translated into Drizzle ORM TypeScript definitions across 15 schema files. Critical implementation areas include: (1) RLS policies for tenant isolation using session variables, (2) exclusion constraints with btree_gist for preventing overlapping bookings, (3) advisory locks for concurrent reservation handling, (4) trigger functions for audit trails and updated_at automation, and (5) seed data generation for development environments.

**Primary recommendation:** Use Drizzle ORM with drizzle-kit for migrations, implement RLS policies via SQL files (not supported natively in Drizzle), use PostgreSQL exclusion constraints for double-booking prevention, and leverage drizzle-seed for deterministic development data.

### Pattern 1: Table Schema Definition with Drizzle

**What:** Define PostgreSQL tables using Drizzle's pgTable with full type inference

**When to use:** Every table in the schema

**Example:**
```typescript

export const companies = pgTable('companies', {
  id: serial('id').primaryKey(),
  uuid: uuid('uuid').defaultRandom().notNull().unique(),
  name: varchar('name', { length: 255 }).notNull(),
  slug: varchar('slug', { length: 255 }).notNull().unique(),
  email: varchar('email', { length: 255 }).notNull(),
  phone: varchar('phone', { length: 50 }),
  subscriptionPlan: varchar('subscription_plan', { length: 20 }).default('free').notNull(),
  industryType: varchar('industry_type', { length: 50 }).default('general').notNull(),
  industryConfig: jsonb('industry_config').default({}).notNull(),
  settings: jsonb('settings').default({}).notNull(),
```

### Pattern 3: Exclusion Constraint for Double-Booking Prevention

**What:** PostgreSQL exclusion constraint using btree_gist to prevent overlapping bookings

**When to use:** Booking table to prevent same employee being double-booked

**Example:**
```sql
-- Source: https://www.cybertec-postgresql.com/en/postgresql-exclusion-constraints-beyond-unique/
-- This must be added via raw SQL in migration or functions/double-booking.sql
-- Drizzle ORM doesn't support EXCLUDE constraints natively yet

CREATE EXTENSION IF NOT EXISTS btree_gist;

-- Prevent overlapping bookings for same employee
ALTER TABLE bookings ADD CONSTRAINT no_overlapping_bookings
  EXCLUDE USING GIST (
    employee_id WITH =,
    tstzrange(start_time, end_time) WITH &&
  )
  WHERE (status NOT IN ('cancelled'));
```

**Note:** PostgreSQL 18 introduces `WITHOUT OVERLAPS` syntax for cleaner temporal constraints, but as of 2026 it's very new. Stick with EXCLUDE USING GIST for broader compatibility.

### Pattern 4: Row Level Security Setup

**What:** Multi-tenant isolation using PostgreSQL RLS with session variables

**When to use:** All tables with company_id column (43 of 47 tables)

**Example:**
```sql
-- Source: https://aws.amazon.com/blogs/database/multi-tenant-data-isolation-with-postgresql-row-level-security/
-- In packages/database/src/rls/functions.sql

-- Helper function to get company_id from session
CREATE OR REPLACE FUNCTION current_company_id() RETURNS INTEGER AS $$
BEGIN
  RETURN NULLIF(current_setting('app.company_id', true), '')::INTEGER;
END;
$$ LANGUAGE plpgsql STABLE SECURITY DEFINER;

-- Helper function to get user role from session
CREATE OR REPLACE FUNCTION current_user_role() RETURNS TEXT AS $$
BEGIN
  RETURN current_setting('app.user_role', true);
END;
$$ LANGUAGE plpgsql STABLE SECURITY DEFINER;

-- In packages/database/src/rls/policies.sql

-- Enable RLS on tenant tables
ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;
ALTER TABLE customers ENABLE ROW LEVEL SECURITY;
ALTER TABLE services ENABLE ROW LEVEL SECURITY;
-- ... all 43 tenant tables

-- Tenant isolation policy (apply to all tenant tables)
CREATE POLICY tenant_isolation_bookings ON bookings
  USING (company_id = current_company_id());

CREATE POLICY tenant_isolation_customers ON customers
  USING (company_id = current_company_id());

-- Admin bypass (optional, for system operations)
CREATE POLICY admin_bypass_bookings ON bookings
  USING (current_user_role() = 'admin');

-- Customer can only see their own bookings
CREATE POLICY customer_own_bookings ON bookings
  FOR SELECT
  USING (
    current_user_role() = 'customer'
    AND customer_id IN (
      SELECT id FROM customers WHERE user_id = current_setting('app.user_id', true)::INTEGER
    )
  );
```

**Critical:** Application middleware must set session variables for EVERY query:
```typescript
// In API middleware (backend segment will implement)
await db.execute(sql`SET LOCAL app.company_id = ${companyId}`);
await db.execute(sql`SET LOCAL app.user_id = ${userId}`);
await db.execute(sql`SET LOCAL app.user_role = ${userRole}`);
```

### Pattern 5: Updated_at Trigger Function

**What:** Automatically update updated_at timestamp on row modification

**When to use:** All 35+ tables with updated_at column

**Example:**
```sql
-- Source: https://emmer.dev/blog/automatic-audit-logging-with-postgresql-triggers/
-- In packages/database/src/functions/updated-at.sql

CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = CURRENT_TIMESTAMP;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Apply to all tables with updated_at column
-- This can be done dynamically in migration:
DO $$
DECLARE
  t TEXT;
BEGIN
  FOR t IN
    SELECT table_name FROM information_schema.columns
    WHERE column_name = 'updated_at' AND table_schema = 'public'
  LOOP
    EXECUTE format('
      CREATE TRIGGER trg_%s_updated_at
      BEFORE UPDATE ON %I
      FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
    ', t, t);
  END LOOP;
END;
$$;
```

### Pattern 6: Drizzle Seed Data with drizzle-seed

**What:** Generate deterministic, realistic seed data for development

**When to use:** Development environment initialization

**Example:**
```typescript
// Source: https://orm.drizzle.team/docs/seed-overview
// packages/database/src/seeds/development.ts
import { drizzle } from 'drizzle-orm/postgres-js';
import { seed } from 'drizzle-seed';
import postgres from 'postgres';
import * as schema from '../schema';

const client = postgres(process.env.DATABASE_URL!);
const db = drizzle(client, { schema });

async function main() {
  // Reset database
  await seed(db, { schema }).reset();

  // Seed with 10 companies (default count)
  await seed(db, { schema }).refine((funcs) => ({
    companies: {
      count: 5,
      columns: {
        name: funcs.company(),
        slug: funcs.slug(),
        email: funcs.email(),
        industryType: funcs.valuesFromArray(['beauty_salon', 'barbershop', 'spa_wellness', 'fitness_gym']),
      },
    },
    users: {
      count: 20,
      with: {
        companyId: 'companies', // Auto-link to companies
        roleId: funcs.valuesFromArray([1, 2, 3, 4]), // admin, owner, employee, customer
      },
    },
    customers: {
      count: 100,
      with: {
        companyId: 'companies',
      },
      columns: {
        name: funcs.fullName(),
        email: funcs.email(),
        phone: funcs.phoneNumber(),
      },
    },
  }));

  console.log('✅ Database seeded successfully');
  await client.end();

main();
```

### Anti-Patterns to Avoid

- **Exposing SERIAL IDs in API responses:** Use UUIDs for public-facing IDs, SERIAL only for internal DB operations
- **Forgetting RLS session variables:** Every query MUST set app.company_id or tenant isolation breaks
- **Using drizzle-kit push in production:** Only for dev. Production uses `db:generate` → `db:migrate` for version control
- **Hand-rolling UUID generation:** Use `uuid().defaultRandom()` or PostgreSQL `uuid_generate_v4()` extension
- **Not using transactions for bookings:** Double-booking prevention requires advisory locks within transactions
- **Ignoring cascade behavior:** ON DELETE CASCADE vs RESTRICT vs SET NULL must match business logic

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Seed data generation | Custom faker scripts with manual FK resolution | drizzle-seed | Respects constraints, deterministic PRNG, auto-resolves relationships, built-in reset |
| Migration system | Custom SQL versioning | drizzle-kit generate + migrate | Type-safe, auto-generates from schema changes, tracks applied migrations |
| Connection pooling | Manual postgres client management | postgres library with config | Built-in pooling, prepared statements, transaction support |
| Timestamp management | Manual NOW() calls | PostgreSQL DEFAULT CURRENT_TIMESTAMP + trigger | Database-level consistency, timezone handling, audit trails |
| Soft delete queries | WHERE deleted_at IS NULL everywhere | Partial indexes + helpers | Performance, consistency, GDPR compliance patterns |

**Key insight:** Database schema complexity explodes with multi-tenancy. Use battle-tested solutions (RLS, exclusion constraints, advisory locks) rather than application-level checks. The database enforces correctness; the app just queries.

## Common Pitfalls

### Pitfall 1: RLS Policy Not Applied, Tenant Data Leaks

**What goes wrong:** Developer forgets to set `app.company_id` session variable before query, RLS policy fails open, returns all companies' data

**Why it happens:** RLS is additive security layer but requires application cooperation. If session variable isn't set, `current_company_id()` returns NULL, policy fails to match, no rows returned (best case) or bypassed with weak policy (worst case)

**How to avoid:**
- Use middleware that ALWAYS sets session variables at connection start
- Test with multiple companies in dev environment
- Add monitoring for queries without company_id
- Use restrictive RLS policies with SECURITY DEFINER functions

**Warning signs:**
- API returning data from wrong company
- Performance regression (full table scans when RLS can't use company_id index)
- Auth tests passing but multi-tenant tests failing

### Pitfall 2: Migration Rollback Confusion

**What goes wrong:** Developer assumes drizzle-kit has automatic rollback/down migrations, discovers it doesn't exist when production migration fails

**Why it happens:** Drizzle ORM philosophy is "forward-only" migrations. Rollback requires manual SQL or new forward migration

**How to avoid:**
- Test migrations thoroughly in staging environment
- For critical changes, write manual rollback SQL before applying
- Use database backups before major migrations
- Consider creating separate rollback.sql files alongside migrations
- Document rollback procedures in migration comments

**Warning signs:**
- No rollback strategy documented
- Migrations tested only in fresh databases, not against production-like data
- Schema changes without corresponding seed data updates

**Current status (2026):** Community actively requesting down migrations (GitHub Issue #2352), but not yet implemented. Workaround: Create new migrations that reverse changes.

### Pitfall 3: Double-Booking Race Condition Despite Constraints

**What goes wrong:** Two concurrent requests create bookings for same employee/time, both succeed despite exclusion constraint

**Why it happens:** Exclusion constraint checks at transaction commit, but without explicit locking, both transactions read "no conflict" simultaneously

**How to avoid:**
- Use advisory locks: `SELECT pg_advisory_xact_lock(hashtext('booking_lock'), employee_id)`
- Combine with exclusion constraint (defense in depth)
- Set SERIALIZABLE isolation level for booking transactions
- Add application-level validation before database insert

**Warning signs:**
- Race condition tests pass but production has conflicts
- High concurrency periods show duplicate bookings
- Database constraint violations in logs but users report success

**Example fix:**
```typescript
// In booking creation (backend will implement):
await db.transaction(async (tx) => {
  // 1. Acquire advisory lock for this employee+date
  await tx.execute(sql`SELECT pg_advisory_xact_lock(
    hashtext('booking_lock'),
    ${employeeId}::INTEGER * 10000 + EXTRACT(DOY FROM ${startTime})::INTEGER
  )`);

  // 2. Check availability
  const conflicts = await tx.select()
    .from(bookings)
    .where(and(
      eq(bookings.employeeId, employeeId),
      notInArray(bookings.status, ['cancelled']),
      sql`${bookings.startTime} < ${endTime}`,
      sql`${bookings.endTime} > ${startTime}`
    ));

  if (conflicts.length > 0) {
    throw new Error('Time slot not available');
  }

  // 3. Insert booking
  const [booking] = await tx.insert(bookings).values({...}).returning();
  return booking;
  // Advisory lock automatically released at transaction end
});
```

### Pitfall 4: View Schema Inference Issues

**What goes wrong:** Drizzle views created from complex queries have incorrect TypeScript types or fail to generate properly

**Why it happens:** Drizzle infers view columns from query builder, but complex queries (joins, aggregates, window functions) can confuse inference

**How to avoid:**
- Explicitly define view columns with types when using raw SQL
- Test view queries independently before wrapping in Drizzle view
- For v_daily_booking_summary and v_customer_metrics, use explicit column definitions
- Materialized views require manual management (not fully supported in drizzle-kit as of 2026)

**Warning signs:**
- TypeScript errors on view queries
- drizzle-kit generate fails with view errors
- View columns nullable when they shouldn't be

**Workaround:** Create views as raw SQL in migration, then create Drizzle view definition with explicit types:
```typescript
export const dailyBookingSummary = pgView('v_daily_booking_summary').as((qb) => {
  return qb.select({
    companyId: bookings.companyId,
    bookingDate: sql<string>`DATE(${bookings.startTime})`,
    totalBookings: sql<number>`COUNT(*)`,
    completed: sql<number>`COUNT(*) FILTER (WHERE ${bookings.status} = 'completed')`,
    cancelled: sql<number>`COUNT(*) FILTER (WHERE ${bookings.status} = 'cancelled')`,
    noShows: sql<number>`COUNT(*) FILTER (WHERE ${bookings.status} = 'no_show')`,
    totalRevenue: sql<number>`SUM(${bookings.price} - ${bookings.discountAmount}) FILTER (WHERE ${bookings.status} = 'completed')`,
  })
  .from(bookings)
  .groupBy(bookings.companyId, sql`DATE(${bookings.startTime})`);
});
```

### Pitfall 5: Seed Data Foreign Key Violation Errors

**What goes wrong:** Seed script fails with FK constraint violations when inserting data in wrong order

**Why it happens:** Manual seed scripts insert tables in arbitrary order, forgetting dependency graph

**How to avoid:**
- Use drizzle-seed which automatically resolves FK dependencies
- If manual seeding, follow dependency order: companies → users/roles → customers → services → employees → bookings
- Wrap seeds in transaction for all-or-nothing behavior
- Use `DEFER CONSTRAINTS` for circular dependencies (rare)

**Warning signs:**
- Seed script works sometimes but fails others
- Different seed execution order on different machines
- FK errors mentioning "violates foreign key constraint"

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|---|---|---|---|
| PostgreSQL SERIAL type | IDENTITY columns (GENERATED ALWAYS AS IDENTITY) | PostgreSQL 10+ | SERIAL is syntactic sugar, IDENTITY is SQL standard, better permissions control |
| Prisma with custom SQL for RLS | Drizzle ORM with raw SQL files | 2023-2024 | Drizzle has better PostgreSQL feature support, Prisma still no native RLS |
| Manual migration versioning | drizzle-kit generate/migrate | Drizzle 0.20+ | Automatic migration generation from schema changes, type-safe |
| Separate seed libraries (@snaplet) | drizzle-seed built-in | Drizzle Seed 2.0 (2024) | Integrated with ORM, automatic FK resolution, deterministic PRNG |
| Exclusion constraints manual SQL | Drizzle schema with custom SQL | Current | Drizzle doesn't support EXCLUDE natively, requires raw SQL in migration |
| View definitions in SQL only | Drizzle pgView with type inference | Drizzle 0.26+ | Type-safe views, but materialized views still need manual handling |

**Deprecated/outdated:**
- **node-postgres pg package directly:** Use postgres (pg wrapper) for modern ESM, better pooling
- **Prisma for PostgreSQL-heavy apps:** Drizzle has surpassed it for PostgreSQL-specific features
- **uuid-ossp extension for UUIDs:** PostgreSQL 13+ has built-in gen_random_uuid(), use `uuid().defaultRandom()`
- **Drizzle Kit push for production:** Only for dev rapid prototyping, production requires generate → migrate workflow

## Open Questions

1. **Materialized View Refresh Strategy**
   - What we know: Drizzle Seed and Drizzle Kit have limited materialized view support, views must be refreshed manually
   - What's unclear: Best pattern for auto-refreshing v_daily_booking_summary and v_customer_metrics (pg_cron? trigger? application-level?)
   - Recommendation: Start with manual refresh via scheduled job, defer auto-refresh to Phase 3 optimization

2. **Partitioning Implementation with Drizzle ORM**
   - What we know: Documentation specifies monthly partitioning for analytics_events and audit_logs
   - What's unclear: Drizzle ORM doesn't have native partitioning syntax, requires raw SQL
   - Recommendation: Create partitioned tables in initial migration via raw SQL, document maintenance in README

3. **Advisory Lock Pattern for High Concurrency**
   - What we know: Advisory locks prevent double-booking race conditions
   - What's unclear: Lock granularity (per-employee, per-employee-date, per-company?) and performance impact
   - Recommendation: Use employee_id * 10000 + day_of_year hash for fine-grained locking, benchmark in Phase 3

4. **Soft Delete vs Hard Delete Strategy**
   - What we know: Documentation shows deleted_at for customers, payments, bookings (GDPR)
   - What's unclear: Which other tables need soft delete? Consistency in query helpers?
   - Recommendation: Add deleted_at to customers, bookings, payments, reviews (user-generated content), use partial indexes

5. **Seed Data Realism Requirements**
   - What we know: drizzle-seed provides deterministic data, @faker-js/faker for customization
   - What's unclear: Level of realism needed (full AI-predicted fields? valid Czech addresses? realistic working hours?)
   - Recommendation: Start with basic drizzle-seed, add custom generators for Czech locale (phone, address) if frontend testing requires
