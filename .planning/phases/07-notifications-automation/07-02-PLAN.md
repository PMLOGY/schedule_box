<objective>
Create the notification worker microservice with BullMQ job queues, delivery channel senders (email/SMS/push), Handlebars template rendering, and default notification templates.

Purpose: This is the core delivery engine. Event consumers (Plan 07-03) will enqueue jobs to these BullMQ queues, and the workers here process them with retry logic, rate limiting, and status tracking. Without this, no notifications can actually be sent.

Output: Complete `services/notification-worker/` package with BullMQ workers for email/SMS/push, Nodemailer email sender, Twilio SMS sender, web-push sender, Handlebars template renderer with Czech locale helpers, and default .hbs templates for booking confirmation, reminder, and review request.
</objective>

**Delivery Channel Senders:**

`services/notification-worker/src/services/email-sender.ts`:
- Create Nodemailer transporter with SMTP config from config.ts
- Enable connection pooling: `pool: true, maxConnections: 5, maxMessages: 100`
- `sendEmail(options: { to, subject, html, from? }): Promise<string>` - sends email, returns messageId
- `injectTrackingPixel(html: string, notificationId: number): string` - injects 1x1 transparent PNG pixel before </body> tag. Pixel URL: `{appUrl}/api/v1/webhooks/email-tracking/open?nid={notificationId}`
- Handle case where SMTP is not configured: log warning and return mock messageId (for dev)

`services/notification-worker/src/services/sms-sender.ts`:
- Create Twilio client from config.ts credentials
- `sendSMS(options: { to, body }): Promise<string>` - sends SMS, returns message SID
- `estimateSMSSegments(body: string): number` - checks if unicode (Czech diacritics), returns segment count. GSM-7: 160 chars, UCS-2: 70 chars.
- Handle case where Twilio is not configured: log warning and return mock SID (for dev)

`services/notification-worker/src/services/push-sender.ts`:
- Configure web-push with VAPID details from config.ts
- `sendPushNotification(subscription: { endpoint, keys: { p256dh, auth } }, payload: { title, body, icon?, url? }): Promise<void>`
- Handle case where VAPID keys not configured: log warning (for dev)

`services/notification-worker/src/services/template-renderer.ts`:
- Template cache: `Map<string, HandlebarsTemplateDelegate>()` for compiled templates
- `renderTemplate(templateString: string, data: Record<string, unknown>): string` - compile with cache, render with data
- `renderTemplateFile(templateName: string, channel: 'email' | 'sms', data: Record<string, unknown>): string` - load from `templates/{channel}/{templateName}.hbs`, render
- Register Handlebars helpers:
  - `formatDate(date, locale='cs-CZ')` - returns localized date string
  - `formatTime(date, locale='cs-CZ')` - returns HH:MM format
  - `formatCurrency(amount, currency='CZK')` - returns formatted currency (cs-CZ Intl)
  - `ifEquals(a, b, options)` - block helper for conditional rendering

`services/notification-worker/src/services/notification-logger.ts`:
- `logNotificationSent(notificationId: number, messageId: string): Promise<void>` - updates notifications table: status='sent', sentAt=now(), metadata includes messageId
- `logNotificationFailed(notificationId: number, error: string): Promise<void>` - updates notifications table: status='failed', errorMessage=error
- `createNotificationRecord(data: { companyId, customerId?, bookingId?, templateId?, channel, recipient, subject?, body, scheduledAt? }): Promise<number>` - inserts into notifications table with onConflictDoNothing for idempotency, returns id
- Uses `@schedulebox/database` with db import and notifications schema

**BullMQ Job Handlers:**

`services/notification-worker/src/jobs/email-job.ts`:
- Interface `EmailJobData`: `{ companyId, bookingId?, customerId?, templateId?, recipient, subject, html, notificationId?, eventId? }`
- `createEmailWorker(redisConnection)` - creates BullMQ Worker on `notification:email` queue:
  - Concurrency: 5
  - Rate limiter: max 100 per 60s (prevent SMTP throttling)
  - stalledInterval: 30000, maxStalledCount: 3
  - Job handler: creates notification record (if notificationId not provided), injects tracking pixel, calls sendEmail, updates notification status to sent (or failed on error + re-throw for retry)
  - On 'completed': log success
  - On 'failed': log error with job ID and attempt number

`services/notification-worker/src/jobs/sms-job.ts`:
- Interface `SmsJobData`: `{ companyId, customerId?, bookingId?, recipient, body, notificationId?, eventId? }`
- `createSmsWorker(redisConnection)` - creates BullMQ Worker on `notification:sms` queue:
  - Concurrency: 3 (SMS more expensive, lower throughput)
  - stalledInterval: 30000, maxStalledCount: 2
  - Job handler: creates notification record, calls sendSMS, updates status
  - Same logging pattern as email worker

`services/notification-worker/src/jobs/push-job.ts`:
- Interface `PushJobData`: `{ companyId, customerId?, subscription: { endpoint, keys }, title, body, icon?, url?, notificationId?, eventId? }`
- `createPushWorker(redisConnection)` - creates BullMQ Worker on `notification:push` queue:
  - Concurrency: 10 (push is fast)
  - stalledInterval: 15000, maxStalledCount: 3
  - Job handler: creates notification record, calls sendPushNotification, updates status
