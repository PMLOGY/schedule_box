Create the event consumer infrastructure and shared types/schemas needed by the entire notification system.

Purpose: All subsequent plans depend on these foundational types and the consumer connection helper. Without a consumer helper in @schedulebox/events, the notification worker cannot subscribe to RabbitMQ. Without Zod schemas, API routes cannot validate input. Without event type definitions, consumers cannot type-check incoming messages.

Output: Consumer connection helper in events package, CloudEvent types for notification/review/payment domains, Zod validation schemas for notification templates, notifications, and automation rules.
</objective>

  <action>
Create `packages/events/src/consumer.ts` with a consumer connection helper that mirrors the publisher pattern (callback-based amqplib API per Phase 5 decision):

1. `createConsumerConnection(options: { url?: string })` - Creates RabbitMQ connection + channel with error/close handlers (mirror publisher.ts pattern). Returns `{ connection, channel }`. Uses RABBITMQ_URL env var with same default as publisher (`amqp://schedulebox:schedulebox@localhost:5672`).

2. `consumeMessages<T>(channel, options: { queueName, exchangeName?, routingKeys, prefetch?, handler: (event: CloudEvent<T>, msg: ConsumeMessage) => Promise<void> })` - Asserts durable queue, binds to routing keys on `schedulebox.events` exchange (default), sets prefetch (default 10), consumes with ACK on success/NACK+requeue on error. Logs consumer startup.

3. `gracefulShutdown(connection, channel)` - Closes channel then connection (mirror closeConnection in publisher.ts).

Create `packages/events/src/events/payment.ts`:
- `PaymentCompletedPayload` interface: `{ paymentUuid, bookingUuid, companyId, amount, currency, gateway, completedAt }`
- `PaymentFailedPayload` interface: `{ paymentUuid, bookingUuid, companyId, gateway, reason, failedAt }`
- `PaymentRefundedPayload` interface: `{ paymentUuid, bookingUuid, companyId, amount, currency, refundedAt }`
- Factory functions: `createPaymentCompletedEvent`, `createPaymentFailedEvent`, `createPaymentRefundedEvent`
- Use `com.schedulebox.payment.*` type prefix, source `payment-service`

Create `packages/events/src/events/review.ts`:
- `ReviewCreatedPayload` interface: `{ reviewUuid, companyId, customerId, bookingId, rating, createdAt }`
- Factory: `createReviewCreatedEvent`
- Use `com.schedulebox.review.created` type, source `review-service`

Create `packages/events/src/events/notification.ts`:
- `NotificationSentPayload` interface: `{ notificationId, companyId, channel, recipient, templateType, sentAt }`
- `NotificationFailedPayload` interface: `{ notificationId, companyId, channel, recipient, error, failedAt }`
- `NotificationOpenedPayload` interface: `{ notificationId, companyId, openedAt }`
- `NotificationClickedPayload` interface: `{ notificationId, companyId, url, clickedAt }`
- Factory functions for each
- Use `com.schedulebox.notification.*` type prefix, source `notification-service`

Update `packages/events/src/index.ts` to re-export all new types and factory functions from the three new event files, plus export consumer helper functions from consumer.ts.

Follow the EXACT same patterns as booking.ts for event types (CloudEvent<Payload>, type aliases, factory functions using createCloudEvent). Follow publisher.ts patterns for consumer.ts (callback-based amqplib, error handlers, logging).
