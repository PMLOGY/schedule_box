**`apps/web/app/api/v1/notification-templates/route.ts`:**
- GET: List templates filtered by companyId, optional type and channel filters. Returns all active templates. Use `db.select().from(notificationTemplates).where(and(eq(notificationTemplates.companyId, companyId), ...filters))`. Paginate with limit/offset.
- POST: Create template. Validate with notificationTemplateCreateSchema. Insert into notificationTemplates with companyId. Note: unique constraint on (company_id, type, channel) - handle 409 conflict.

**`apps/web/app/api/v1/notification-templates/[id]/route.ts`:**
- GET: Get single template by ID, scoped to company.
- PUT: Update template. Validate with notificationTemplateUpdateSchema. Update by id + companyId.
- DELETE: Delete template by id + companyId.
- UUID-to-ID mapping: templates table uses SERIAL id (no uuid column). Accept numeric ID in route param. Validate with parseInt, return 400 if NaN.

**`apps/web/app/api/v1/notification-templates/[id]/preview/route.ts`:**
- POST: Accepts `{ testData: Record<string, unknown> }`, renders template body with Handlebars using provided test data, returns rendered HTML/text. Import Handlebars directly in API route (lightweight, no need for full worker). This lets admins preview templates with sample data before saving.

**`apps/web/app/api/v1/notifications/route.ts`:**
- GET: List notifications for company with filters (channel, status, customerId, dateFrom, dateTo). Validate query with notificationListQuerySchema. Paginate. Order by createdAt DESC.

**`apps/web/app/api/v1/notifications/[id]/route.ts`:**
- GET: Get single notification by ID, scoped to company. Include related template info if templateId exists.

**`apps/web/app/api/v1/webhooks/email-tracking/open/route.ts`:**
- GET (NO AUTH - public webhook): Read `nid` query param. If valid, update notification: set status='opened', openedAt=now() WHERE id=nid AND openedAt IS NULL (first open only). Return 1x1 transparent PNG. Use base64-encoded minimal PNG. Set Cache-Control: no-cache, no-store.

**`apps/web/app/api/v1/webhooks/email-tracking/click/route.ts`:**
- GET (NO AUTH - public webhook): Read `nid` and `url` query params. Update notification: set status='clicked', clickedAt=now() WHERE id=nid AND clickedAt IS NULL. Redirect (302) to the target URL. Validate URL starts with http/https to prevent open redirect.

**`apps/web/app/api/v1/webhooks/push/register/route.ts`:**
- POST (requires auth): Accepts push subscription object `{ endpoint, keys: { p256dh, auth } }`. Store in user's record or a push_subscriptions metadata (use users table metadata JSONB field, or create simple storage in notification metadata). Returns 201.

**`apps/web/app/api/v1/automation/rules/route.ts`:**
- GET: List automation rules for company. Optional filters: triggerType, isActive. Paginate. Use automationRuleListQuerySchema for query validation.
- POST: Create rule. Validate with automationRuleCreateSchema. Insert into automationRules with companyId. Return 201 with created rule (include uuid for frontend reference).

**`apps/web/app/api/v1/automation/rules/[id]/route.ts`:**
- GET: Get rule by UUID (automationRules has uuid column). Query: `where: and(eq(automationRules.uuid, id), eq(automationRules.companyId, companyId))`.
- PUT: Update rule by UUID + companyId. Validate with automationRuleUpdateSchema.
- DELETE: Delete rule by UUID + companyId.

**`apps/web/app/api/v1/automation/rules/[id]/toggle/route.ts`:**
- POST: Toggle rule active status. Read current isActive, flip it. Update: `set({ isActive: !currentRule.isActive, updatedAt: new Date() })`. Return updated rule with new isActive state. This is a convenience endpoint so the UI can toggle with a single click.

**`apps/web/app/api/v1/automation/logs/route.ts`:**
- GET: List automation execution logs for company. Join automationLogs with automationRules to get rule name. Filter by ruleId (optional), status (optional). Paginate. Order by createdAt DESC. Only return logs for rules belonging to the user's company (join on automationRules.companyId = companyId).

Permission: All automation endpoints use PERMISSIONS.SETTINGS_UPDATE (or define a new PERMISSIONS.AUTOMATION_MANAGE if more granular control needed - check existing PERMISSIONS constant first and use what's available).

UUID handling: automationRules has a uuid column, so use UUID for route params (matches the API convention of never exposing SERIAL IDs). Parse route param as UUID string, query by uuid column.
