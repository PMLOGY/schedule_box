<objective>
Create the reminder notification scheduler and automation rule execution engine that process time-based triggers and rule-based automation.

Purpose: Reminders (24h and 2h before appointment) are the most impactful notification type for reducing no-shows. The automation engine allows business owners to create custom workflows (e.g., "when booking completed, wait 2 hours, send review request") without code changes. Both systems run as background processes within the notification worker.

Output: BullMQ repeatable job for reminder scheduling, automation rule execution engine triggered by domain events, execution logging for debugging/audit.
</objective>

Create `services/notification-worker/src/schedulers/reminder-scheduler.ts`:

`startReminderScheduler(emailQueue: Queue, smsQueue: Queue, redisConnection)`:

1. Create a BullMQ Queue named `notification:reminders` for the repeatable job scheduling.

2. Add a repeatable job that runs every 15 minutes:
   ```
   remindersQueue.add('scan-reminders', {}, {
     repeat: { every: 15 * 60 * 1000 },
     jobId: 'reminder-scanner'
   })
   ```

3. Create a BullMQ Worker for `notification:reminders` that on each execution:

   a. Calculate time windows:
      - 24h window: `now + 23h45m` to `now + 24h15m` (30-min window around 24h mark, accounts for scanner running every 15min)
      - 2h window: `now + 1h45m` to `now + 2h15m` (30-min window around 2h mark)

   b. Query bookings in both windows:
      ```sql
      SELECT b.*, c.email, c.phone, c.first_name, s.name as service_name, e.name as employee_name
      FROM bookings b
      JOIN customers c ON b.customer_id = c.id
      JOIN services s ON b.service_id = s.id
      LEFT JOIN employees e ON b.employee_id = e.id
      WHERE b.start_time BETWEEN $windowStart AND $windowEnd
      AND b.status IN ('confirmed', 'pending')
      AND b.company_id IS NOT NULL
      ```
      Use Drizzle ORM: `db.select(...).from(bookings).innerJoin(customers, ...).innerJoin(services, ...).leftJoin(employees, ...).where(and(between(bookings.startTime, windowStart, windowEnd), inArray(bookings.status, ['confirmed', 'pending'])))`

   c. For each booking found, check if reminder already sent:
      - Query notifications table: WHERE bookingId = booking.id AND channel = 'email' AND subject LIKE '%pripomenut%' AND status != 'failed'
      - If reminder already exists, skip (idempotency)

   d. For bookings needing reminder:
      - Determine reminder type: '24h' or '2h' based on which window matched
      - Fetch booking_reminder template for company (or use default file template)
      - Template data: customer_name, service_name, booking_date, booking_time, employee_name, company_name
      - Enqueue email job with jobId = `reminder-${booking.id}-${reminderType}` (deduplication)
      - If customer has phone, also enqueue SMS reminder with similar jobId pattern

   e. Log: "Scanned reminders: {24h_count} x 24h, {2h_count} x 2h reminders enqueued"

4. CRITICAL timezone handling: All calculations in UTC. bookings.startTime is TIMESTAMPTZ (stored as UTC). Do NOT mix local and UTC times. The template renderer handles formatting to cs-CZ locale for display.

Create `services/notification-worker/src/schedulers/automation-engine.ts`:

1. `processAutomationRules(eventType: string, eventData: Record<string, unknown>, companyId: number, eventId: string, queues: { emailQueue, smsQueue, pushQueue })`:
   - Map CloudEvent type to automation triggerType:
     - `com.schedulebox.booking.created` -> `booking_created`
     - `com.schedulebox.booking.confirmed` -> `booking_confirmed`
     - `com.schedulebox.booking.completed` -> `booking_completed`
     - `com.schedulebox.booking.cancelled` -> `booking_cancelled`
     - `com.schedulebox.booking.no_show` -> `booking_no_show`
     - `com.schedulebox.payment.completed` -> `payment_received`
     - `com.schedulebox.review.created` -> `review_received`
   - If event type doesn't map to a trigger, return early (prevent loop: notification.* events are NOT automation triggers)

   - Query active automation rules:
     ```
     db.select().from(automationRules)
       .where(and(
         eq(automationRules.companyId, companyId),
         eq(automationRules.triggerType, mappedTriggerType),
         eq(automationRules.isActive, true)
       ))
     ```

   - For each matching rule:
     a. Create automation_log record with status='pending'
     b. Determine action based on rule.actionType:
        - `send_email`: Fetch template from rule.actionConfig.templateId, render with event data, enqueue to emailQueue
        - `send_sms`: Same for SMS, enqueue to smsQueue
        - `send_push`: Fetch template from rule.actionConfig.templateId, look up customer push subscription (same helper pattern as in 07-03 consumers), enqueue to pushQueue with subscription data. If no push subscription found for the customer, log as 'skipped' with reason 'no push subscription'
        - Other action types: Log as 'skipped' (not implemented in Phase 7)
     c. Apply delay: If rule.delayMinutes > 0, add `delay: rule.delayMinutes * 60 * 1000` to BullMQ job options
     d. jobId = `auto-${rule.id}-${eventId}` for deduplication
     e. Update automation_log: status='executed', executedAt=now()
     f. On error: Update automation_log: status='failed', errorMessage=error.message

2. `TRIGGER_TYPE_MAP` constant object mapping CloudEvent types to automation trigger types. Only domain events (booking.*, payment.*, review.*) are mapped. notification.* and automation.* are explicitly excluded to prevent infinite loops.

3. Error handling: Each rule processes independently. One failing rule does not prevent others from executing. Log errors per rule.
