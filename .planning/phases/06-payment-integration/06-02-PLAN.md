       processedWebhooks = pgTable('processed_webhooks', {
         eventId: text('event_id').primaryKey(),       // Gateway's unique event/transaction ID
         gatewayName: varchar('gateway_name', { length: 20 }).notNull(),  // 'comgate' | 'qrcomat'
         paymentId: integer('payment_id'),              // FK to payments table (nullable, may not exist yet)
         status: varchar('status', { length: 20 }).notNull().default('processing'),  // 'processing' | 'completed' | 'failed'

         completedAt: timestamp('completed_at', { withTimezone: true }),
       })

       - Add index on (gatewayName, processedAt) for cleanup queries

       - Run `pnpm --filter @schedulebox/database exec drizzle-kit push` or the migration runner to apply

<task type="auto">
  <name>Task 2: Payment Service Foundation</name>
  <files>
    apps/web/app/api/v1/payments/service.ts
  </files>
  <action>
    Create the payment service layer with reusable business logic functions. Follow the pattern from existing service files (e.g., how customers/route.ts uses inline service logic, but extract to dedicated service file for reuse across multiple payment routes).

    **apps/web/app/api/v1/payments/service.ts** — Create these functions:

    1. `createPaymentRecord(params)` — Insert new payment record:
       - Params: { companyId, bookingId, customerId, amount, currency, gateway, gatewayTransactionId? }
       - Uses Drizzle insert into payments table
       - Returns created payment with uuid
       - Validates booking exists and belongs to company (SELECT with company_id filter)

    2. `updatePaymentStatus(paymentId, status, extra?)` — Update payment status:
       - Uses SELECT FOR UPDATE within transaction to prevent race conditions
       - Params: paymentId (internal), status, extra (optional: paidAt, refundAmount, refundReason, refundedAt, gatewayTransactionId, gatewayResponse)
       - Validates status transition is valid (pending->paid, pending->failed, paid->refunded, paid->partially_refunded)
       - Returns updated payment

    3. `generateInvoiceNumber(companyId, tx?)` — Generate sequential invoice number:
       - Format: YYYY-NNNN (year + zero-padded sequential number)
       - Uses SELECT MAX(invoice_number) FROM invoices WHERE company_id = $1 AND invoice_number LIKE 'YYYY-%' within transaction
       - Returns next number (e.g., '2026-0001', '2026-0002')
       - If no invoices exist for year, starts at '2026-0001'
       - MUST be called within a transaction (accepts Drizzle tx parameter) to prevent race conditions

    4. `findPaymentByGatewayTx(gateway, gatewayTransactionId)` — Lookup payment by gateway transaction:
       - Uses the idx_payments_gateway_tx composite index
       - Returns payment or null

    5. `checkWebhookIdempotency(eventId, gatewayName, payload)` — Idempotency check:
       - Try INSERT into processed_webhooks with eventId
       - If 23505 (unique_violation), return { alreadyProcessed: true }
       - If success, return { alreadyProcessed: false }
       - On error, re-throw

    6. `markWebhookCompleted(eventId)` — Update processed webhook status:
       - UPDATE processed_webhooks SET status = 'completed', completedAt = now() WHERE event_id = $1

    Import database from @schedulebox/database. Import AppError and error classes from @schedulebox/shared. Use existing db instance from the project (check how other routes import it — likely from @schedulebox/database or @/lib/db).

    IMPORTANT: All queries that read payment data MUST filter by companyId for tenant isolation, EXCEPT webhook idempotency (global).
  </action>
