  <action>
    Create the event handler functions for the booking-payment SAGA choreography pattern.

    NOTE: In the current architecture (Phase 5 Plan 02), event publishing is fire-and-forget. There is no RabbitMQ consumer infrastructure yet (consumers will be added in Phase 7 for notifications). For MVP, these SAGA handlers will be called DIRECTLY from the webhook handler (Plan 06-03) and from the payment timeout function, NOT via RabbitMQ consumers. This is a pragmatic approach: the event-driven SAGA logic is correct, but invocation is synchronous for now. When Phase 7 adds proper consumers, these handlers become the consumer callbacks with zero code changes.

    **apps/web/app/api/v1/payments/saga/booking-payment-handlers.ts:**

    1. `handlePaymentCompleted(data: PaymentCompletedPayload)` — Happy path:
       - Import PaymentCompletedPayload type from @schedulebox/events
       - Find booking by bookingUuid (use db.select from bookings table)
       - Use transaction with SELECT FOR UPDATE on booking row to prevent race conditions
       - Verify booking status is 'pending' (if already 'confirmed', idempotent — return success)
       - If booking is 'cancelled', log warning and return (late payment after timeout — would need manual reconciliation)
       - Update booking status to 'confirmed' with confirmedAt = new Date() (note: bookings table doesn't have confirmedAt column — use updatedAt, or simply set status='confirmed')
       - Publish createBookingConfirmedEvent from @schedulebox/events
       - Log: "[SAGA] Booking {uuid} confirmed after payment {paymentUuid}"

    2. `handlePaymentFailed(data: PaymentFailedPayload)` — Compensation:
       - Find booking by bookingUuid
       - Use transaction with SELECT FOR UPDATE
       - Verify booking status is 'pending' (if already 'cancelled', idempotent — return)
       - If booking is 'confirmed', do NOT cancel (payment succeeded elsewhere, race condition)
       - Update booking: status='cancelled', cancelledAt=new Date(), cancellationReason=data.reason, cancelledBy='system'
       - Publish createBookingCancelledEvent with cancelledBy='system', reason=data.reason
       - Log: "[SAGA] Booking {uuid} cancelled — payment failed: {reason}"

    3. `handlePaymentExpired(data: PaymentExpiredPayload)` — Timeout compensation:
       - Find booking by bookingUuid
       - Use transaction with SELECT FOR UPDATE
       - Same logic as handlePaymentFailed but with reason='payment_timeout'
       - Update booking: status='cancelled', cancelledBy='system', cancellationReason='Payment timeout (30 minutes)'
       - Publish createBookingCancelledEvent
       - Log: "[SAGA] Booking {uuid} cancelled — payment expired"

    IMPORTANT: All handlers must be idempotent. If called twice for same event, second call should be a no-op. Check current booking status before modifying.

    After creating this file, UPDATE the Comgate webhook handler (apps/web/app/api/v1/webhooks/comgate/route.ts) to call these handlers:
    - After publishing payment.completed event, also call handlePaymentCompleted(eventData)
    - After publishing payment.failed event, also call handlePaymentFailed(eventData)
    This ensures the SAGA flows execute immediately (not waiting for async consumer that doesn't exist yet).

  <action>
    Create the payment timeout mechanism for abandoned payments.

    **apps/web/app/api/v1/payments/saga/payment-timeout.ts:**

    1. `expirePendingPayments(timeoutMinutes?: number)` — Expire old pending payments:
       - Default timeout: 30 minutes (configurable via parameter or PAYMENT_TIMEOUT_MINUTES env var)
       - Query: SELECT * FROM payments WHERE status = 'pending' AND created_at < NOW() - INTERVAL '{timeout} minutes'
       - For each expired payment:
         a. Update payment status to 'expired' (note: 'expired' is not in current CHECK constraint — need to add it OR use 'failed' status with gatewayResponse noting timeout)
         b. Since payments.status CHECK constraint only allows 'pending', 'paid', 'failed', 'refunded', 'partially_refunded', use status='failed' with gatewayResponse = { reason: 'payment_timeout', expiredAt: new Date().toISOString() }
         c. Publish createPaymentExpiredEvent
         d. Call handlePaymentExpired to trigger booking cancellation
       - Return count of expired payments
       - Log: "[Payment Timeout] Expired {count} pending payments older than {timeout} minutes"

    2. `createPaymentTimeoutEndpoint()` — API route handler for manual/cron invocation:
       - This function can be called by:
         a. A cron job (scheduled via external scheduler or Kubernetes CronJob)
         b. Manual admin trigger (POST /api/v1/payments/expire-pending)
       - Returns { expired_count: number, payments: Array<{ paymentUuid, bookingUuid }> }

    3. Create **apps/web/app/api/v1/payments/expire-pending/route.ts** — POST handler:
       - Protected endpoint (requiresAuth: true, owner/admin only)
       - Calls expirePendingPayments()
       - Returns count of expired payments
       - This allows manual triggering and cron job integration

    Add PAYMENT_TIMEOUT_MINUTES=30 to .env.example.

    NOTE: For MVP, this timeout is triggered manually or by cron. A proper scheduled job infrastructure (e.g., node-cron, Kubernetes CronJob) will be set up in Phase 15 (DevOps). For now, the endpoint exists and can be called via cron or manual admin action.
