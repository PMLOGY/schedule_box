<task type="auto">
  <name>Task 1: Invoice Generation Module</name>
  <files>
    apps/web/app/api/v1/invoices/generate.ts
  </files>
  <action>
    Install PDFKit and create invoice generation logic.

    1. Install pdfkit in apps/web:
       ```
       pnpm --filter @schedulebox/web add pdfkit
       pnpm --filter @schedulebox/web add -D @types/pdfkit
       ```
       NOTE: Using pdfkit directly instead of invoice-pdfkit (research suggestion). pdfkit is battle-tested (29M+ weekly downloads), gives full control over Czech invoice formatting, and avoids dependency on a less-maintained wrapper library.

    2. **apps/web/app/api/v1/invoices/generate.ts:**

       `createInvoiceForPayment(paymentId, companyId, tx?)` — Create invoice record:
       - Query payment with booking + customer + service details (JOIN)
       - Generate invoice number using generateInvoiceNumber(companyId, tx) from payment service
       - Calculate tax: amount * 0.21 for 21% Czech VAT (standard rate)
       - Insert into invoices table: { companyId, paymentId, customerId, invoiceNumber, amount: payment.amount, taxAmount, currency, status: 'issued', issuedAt: new Date() }
       - Return created invoice

       `generateInvoicePDF(invoiceId, companyId)` — Generate PDF binary:
       - Query invoice with related payment, booking, customer, service, company data
       - Create PDFKit document with A4 page size
       - Layout (Czech accounting standard):
         - Header: Company name, ICO (company ID), DIC (VAT ID), address (from company settings or companies table)
         - "FAKTURA" (Invoice) title with invoice number
         - Customer section: Name, email, address (if available)
         - Invoice details: Issue date, due date (issue date + 14 days), variable symbol (payment ID)
         - Line items table:
           | Popis (Description) | Mnozstvi (Qty) | Cena bez DPH (Price excl. VAT) | DPH (VAT) | Cena s DPH (Price incl. VAT) |
           - Service name as line item, quantity 1, price from booking
         - Totals: Zaklad dane (Tax base), DPH 21%, Celkem (Total)
         - Payment info: Payment method (gateway name), Transaction ID
         - Footer: "Vystaveno v systemu ScheduleBox" (Generated by ScheduleBox)
       - Return PDF as Buffer (use doc.pipe to buffer, not to file)

       Use PDFKit with Czech characters support. PDFKit supports UTF-8 by default with the built-in Helvetica font. For proper Czech diacritics, the built-in fonts handle most cases. If diacritics fail, embed a custom font file (but try built-in first).

    IMPORTANT: PDF generation should work without external services (no Puppeteer, no browser). PDFKit runs in pure Node.js.
  </action>
  <verify>
    TypeScript compilation passes.
    Both functions exported: createInvoiceForPayment, generateInvoicePDF.
    PDFKit installed in apps/web/package.json.
  </verify>
  <done>
    Invoice creation generates sequential YYYY-NNNN number per company. PDF generation produces A4 document with Czech VAT-compliant layout using PDFKit. No external service dependencies.
  </done>

<task type="auto">
  <name>Task 2: Invoice PDF Download Endpoint</name>
  <files>
    apps/web/app/api/v1/invoices/[id]/pdf/route.ts
  </files>
  <action>
    Create the invoice PDF download endpoint.

    **apps/web/app/api/v1/invoices/[id]/pdf/route.ts** — GET handler:

    1. Use createRouteHandler with requiresAuth: true
    2. Extract invoice UUID from path params ([id] = UUID)
    3. Find invoice by UUID, verify it belongs to user's company (tenant isolation)
    4. If invoice not found, return 404
    5. Call generateInvoicePDF(invoice.id, companyId) to get PDF buffer
    6. Return PDF as binary response:
       ```typescript
       return new Response(pdfBuffer, {
         status: 200,
         headers: {
           'Content-Type': 'application/pdf',
           'Content-Disposition': `attachment; filename="faktura-${invoice.invoiceNumber}.pdf"`,
           'Content-Length': pdfBuffer.length.toString(),
         },
       });
       ```
    7. Use Czech filename "faktura-{number}.pdf" for download

    NOTE: For MVP, PDF is generated on-the-fly each time. Future optimization: store generated PDF in Cloudflare R2 and serve from there (update invoice.pdfUrl). For now, on-the-fly generation is fine for the expected volume.
  </action>
  <verify>
    TypeScript compilation passes.
    Route exports GET handler.
    Response returns application/pdf content type.
  </verify>
  <done>
    GET /api/v1/invoices/{id}/pdf returns downloadable PDF with Czech filename. PDF generated on-the-fly with company, customer, and payment details.
  </done>

<task type="auto">
  <name>Task 3: Payment Refund Endpoint</name>
  <files>
    apps/web/app/api/v1/payments/[id]/refund/route.ts
  </files>
  <action>
    Create the refund processing endpoint.

    **apps/web/app/api/v1/payments/[id]/refund/route.ts** — POST handler:

    1. Use createRouteHandler with requiresAuth: true, requiredPermissions for admin/owner role (only owner can process refunds)
    2. Validate request body with paymentRefundSchema from @schedulebox/shared: { amount? (number, partial refund), reason (string, required) }
    3. Extract payment UUID from path params
    4. Find payment by UUID, verify it belongs to user's company
    5. Validate refund is possible:
       - Payment status must be 'paid' (cannot refund pending/failed/already-refunded)
       - If amount specified, it must be <= (payment.amount - payment.refundAmount) for partial refund
       - If no amount specified, refund full remaining amount
    6. Use transaction with SELECT FOR UPDATE on payment row:
       a. For Comgate gateway: call refundComgatePayment(payment.gatewayTransactionId, refundAmountInHellers)
          - If Comgate refund fails, throw AppError with PAYMENT_REFUND_FAILED
       b. For QR/cash/bank_transfer/gift_card: no external API call needed (manual reconciliation)
       c. Calculate new refund total: existingRefundAmount + thisRefundAmount
       d. Determine new status: if newRefundTotal >= payment.amount then 'refunded' else 'partially_refunded'
       e. Update payment: status, refundAmount = newRefundTotal, refundReason = reason, refundedAt = new Date(), gatewayResponse append refund info
    7. Publish createPaymentRefundedEvent
    8. Return updated payment with status 200

    NOTE on refund idempotency: The endpoint is NOT inherently idempotent (multiple calls = multiple refunds). The UI should disable the refund button after successful refund. The database prevents over-refunding via the amount check in step 5.
  </action>
  <verify>
    TypeScript compilation passes.
    Route exports POST handler.
    Refund validates amount doesn't exceed original payment.
  </verify>
  <done>
    POST /api/v1/payments/{id}/refund processes full or partial refunds. Comgate refunds call gateway API. Non-Comgate refunds are recorded locally. Over-refunding prevented by amount validation.
  </done>
