  <action>
    Install dependencies and create QR payment generation module.

    1. Install packages in apps/web:
       ```
       pnpm --filter @schedulebox/web add qrcode
       pnpm --filter @schedulebox/web add -D @types/qrcode
       ```
       NOTE: Research recommended @spayd/core but it may not be well-maintained. Implement SPD format manually — the format is simple and well-documented (https://qr-platba.cz/pro-vyvojare/specifikace-formatu/). This avoids a potentially unmaintained dependency for what is essentially a string concatenation task.

    2. **apps/web/app/api/v1/payments/qr-payment/client.ts:**

       `generateSPDString(params)` — Generate SPD format string:
       - Params: { iban: string, amount: number, currency: string, variableSymbol: string, message: string }
       - SPD format: `SPD*1.0*ACC:{iban}*AM:{amount}*CC:{currency}*X-VS:{variableSymbol}*MSG:{message}`
       - Validate IBAN format (starts with CZ or SK, 24 chars for CZ, 24 for SK)
       - Amount formatted to 2 decimal places
       - Message truncated to 60 chars (SPD spec limit)
       - Variable symbol max 10 digits
       - Return the SPD string

       `generateCzechQRPayment(params)` — Generate QR code image:
       - Params: { iban, amount, currency, variableSymbol, message }
       - Call generateSPDString to get SPD string
       - Use qrcode.toDataURL(spdString, { errorCorrectionLevel: 'M', width: 300 }) for base64 PNG
       - Return { spdString, qrCodeBase64: dataUrl }

    Company IBAN will be stored in company settings. For now, read from environment variable COMPANY_DEFAULT_IBAN or from company_settings table if available.
  </action>

<task type="auto">
  <name>Task 2: QR Payment Generate Endpoint</name>
  <files>
    apps/web/app/api/v1/payments/qr-payment/generate/route.ts
  </files>
  <action>
    Create the QR payment generation API route.

    **apps/web/app/api/v1/payments/qr-payment/generate/route.ts** — POST handler:

    1. Use createRouteHandler with requiresAuth: true, requiredPermissions for payment/booking access (owner or employee can generate QR for on-site customer)
    2. Validate request body with qrPaymentGenerateSchema from @schedulebox/shared (expects booking_id)
    3. Find booking by ID, verify it belongs to user's company
    4. Verify booking status allows payment (status === 'pending' or 'confirmed')
    5. Check if a pending payment already exists for this booking — if so, return existing QR code (idempotent)
    6. Get company IBAN from company settings (query company_settings table or use a default). If no IBAN configured, return error: "Company bank account (IBAN) not configured. Please update company settings."
    7. Generate variable symbol from booking ID (use the internal serial ID zero-padded to 10 digits, e.g., "0000001234")
    8. Call generateCzechQRPayment with:
       - iban: company IBAN
       - amount: parseFloat(booking.price)
       - currency: booking.currency || 'CZK'
       - variableSymbol: padded booking ID
       - message: "ScheduleBox #{booking.uuid.slice(0,8)}"
    9. Create payment record via createPaymentRecord with gateway='qrcomat', no gatewayTransactionId (QR payment doesn't have one upfront — the variable symbol IS the reference)
    10. Publish payment.initiated event
    11. Return { qr_code_base64: qrCodeBase64, spd_string: spdString, variable_symbol: variableSymbol } with status 200

    NOTE on QR payment confirmation: Unlike Comgate, QR payments don't have automatic webhook confirmation. The payment confirmation happens when the bank transfer arrives (which can take minutes to days). For MVP, the admin will manually mark the payment as paid via the payment update endpoint (Plan 06-07). Future phases can add bank statement import or FIO Bank API integration.
  </action>
