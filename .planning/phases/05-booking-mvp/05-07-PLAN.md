Create `apps/web/lib/booking/time-blocking.ts`:

Two approaches for blocking time, both supported:

**Approach 1: Full-day blocking via schedule overrides (vacation, day off)**
The existing schedule_overrides API (Phase 3, plan 03-07) already handles this:
- POST /api/v1/employees/:id/schedule-overrides with `{ date, is_day_off: true }` creates a day off
- POST /api/v1/employees/:id/schedule-overrides with `{ date, start_time, end_time, is_day_off: false }` creates modified hours
- The availability engine in Plan 05-03 already respects these overrides

No new code needed for this approach — just document that it works.

**Approach 2: Partial-day blocking via blocked bookings (specific time slots)**
For blocking specific time ranges (e.g., "lunch break 12:00-13:00" or "equipment maintenance 14:00-16:00"):

1. `blockTimeSlot(input: { employeeId: number; startTime: Date; endTime: Date; reason: string }, companyId: number): Promise<Booking>`
   - Insert a booking with:
     - `source: 'admin'`
     - `status: 'confirmed'` (so it blocks availability)
     - `customerId: null` (use a sentinel or the admin's own ID) — WAIT: customerId is NOT NULL in schema. Use a system/admin customer record. Alternative: create a special "blocked" entry in a different way.
   - REVISED APPROACH: Since bookings table requires customerId (NOT NULL), blocked time slots should be implemented differently:
     - Option A: Use working_hours_overrides with modified start/end times
     - Option B: Create internal "blocked slots" by inserting into bookings with a designated "system" customer
   - Use Option A (schedule overrides) for cleaner design: Create a working_hours_override for the specific date with the employee's available hours EXCLUDING the blocked period. For example, if working 9:00-17:00 and blocking 12:00-13:00, create two overrides: 9:00-12:00 and 13:00-17:00.
   - This is complex. SIMPLER: Insert into bookings table using the company owner's customer record (or first customer) as a placeholder. Mark with `internalNotes: 'BLOCKED_SLOT'` and `source: 'admin'`. The availability engine already excludes non-cancelled bookings.

   Final implementation:
   - Query or create a "system blocked" customer for this company (name: "Blocked Time", email: null, with a tag "system")
   - Insert booking: `{ companyId, customerId: systemCustomerId, serviceId: null (nullable? NO - required), ... }`
   - PROBLEM: serviceId is also NOT NULL. Need a "blocked" service.

   SIMPLEST CORRECT APPROACH: Use working_hours_overrides to split available hours around blocked periods. This naturally integrates with the availability engine.

   `blockTimeSlot(input: { employeeId: number; date: string; blockStartTime: string; blockEndTime: string; reason?: string }, companyId: number)`:
   - Get employee's working hours for that date (regular or override)
   - If employee has hours 9:00-17:00 and block is 12:00-13:00:
     - Delete existing override for that date (if any)
     - Create override(s) for remaining available periods: [9:00-12:00] and [13:00-17:00]
     - Store reason in override's notes/reason field (if no field exists, use the override pattern)
   - If block covers entire working day, create is_day_off=true override

2. `unblockTimeSlot(overrideId: number, companyId: number)`:
   - Delete the specific schedule override
   - Restore original working hours (delete the split overrides, original regular hours will apply)

3. `listBlockedSlots(employeeId: number, dateFrom: string, dateTo: string, companyId: number)`:
   - Query working_hours_overrides for date range
   - Compare with regular working_hours to identify "gaps" that represent blocked time
   - Return list of blocked periods with dates and times

Create `apps/web/app/api/v1/bookings/block/route.ts`:

**POST**: Create a time block
- `requiresAuth: true`, permissions: `['bookings.create']`
- Body: `{ employee_id: number, date: string, start_time: string, end_time: string, reason?: string }`
- Validate: end_time > start_time, date is not in the past
- Call `blockTimeSlot(input, companyId)`
- Return 201 with block details

**GET**: List blocked slots
- Query params: `employee_id`, `date_from`, `date_to`
- Call `listBlockedSlots(...)`
- Return array of blocked periods

**DELETE**: Remove a time block (by override ID)
- Call `unblockTimeSlot(overrideId, companyId)`
- Return 204

Note: The existing `POST /api/v1/employees/:id/schedule-overrides` from Phase 3 already handles full-day blocks (vacations). This endpoint handles partial-day blocks within working hours.
