Create `apps/web/lib/booking/booking-transitions.ts`:

Define valid state transitions:
```typescript
const VALID_TRANSITIONS: Record<string, string[]> = {
  pending: ['confirmed', 'cancelled', 'expired'],
  confirmed: ['completed', 'cancelled', 'no_show'],
};
```

Helper `validateTransition(currentStatus: string, newStatus: string): void` — throws AppError 422 if invalid transition.

1. **confirmBooking(bookingId: number, companyId: number): Promise<Booking>**
   - Get booking (404 if not found), validate transition pending->confirmed
   - UPDATE status='confirmed', updatedAt=NOW()
   - Publish `booking.confirmed` event
   - Return updated booking

2. **cancelBooking(bookingId: number, input: { reason?: string }, context: { companyId: number; userId: number; userRole: string }): Promise<Booking>**
   - Get booking (404 if not found), validate transition from pending or confirmed
   - **Cancellation policy check**: If caller is 'customer' role:
     - Get service's `cancellationPolicyHours` (default 24)
     - If booking startTime is within cancellationPolicyHours from now, throw AppError 403 with `'CANCELLATION_POLICY'` code and message: "Cannot cancel within {hours} hours of appointment"
   - Admin/employee roles bypass cancellation policy
   - UPDATE: status='cancelled', cancelledAt=NOW(), cancellationReason=reason, cancelledBy=(role-based: 'customer'|'employee'|'admin')
   - Publish `booking.cancelled` event
   - Return updated booking

3. **completeBooking(bookingId: number, companyId: number): Promise<Booking>**
   - Get booking, validate transition confirmed->completed
   - UPDATE status='completed', updatedAt=NOW()
   - Publish `booking.completed` event

4. **markNoShow(bookingId: number, companyId: number): Promise<Booking>**
   - Get booking, validate transition confirmed->no_show
   - UPDATE status='no_show', updatedAt=NOW()
   - Publish `booking.no_show` event
   - (Customer's no_show_count increment is handled by DB trigger from Phase 2)

5. **rescheduleBooking(bookingId: number, input: { start_time: string; employee_id?: number }, context: { companyId: number; userId: number }): Promise<Booking>**
   - Get booking (404 if not found). Must be pending or confirmed.
   - Save old times for event payload
   - Inside `db.transaction()`:
     a. Determine new employee (use input.employee_id or keep existing)
     b. Get service for duration calculation
     c. Calculate new endTime
     d. SELECT FOR UPDATE on employee row
     e. Check no conflicting bookings at new time (exclude current booking from conflict check!)
     f. If conflict: throw AppError 409 SLOT_TAKEN
     g. UPDATE startTime, endTime, employeeId (if changed), updatedAt
   - Publish `booking.rescheduled` event with old and new times
   - Return updated booking

Create `apps/web/lib/booking/booking-expiration.ts`:

**expirePendingBookings(): Promise<number>**
- Per research: simple cron approach for MVP
- Execute: `UPDATE bookings SET status='expired', updated_at=NOW() WHERE status='pending' AND created_at < NOW() - INTERVAL '30 minutes' RETURNING id, uuid, company_id`
- For each expired booking, publish `booking.cancelled` event with cancelledBy='system' and reason='Booking expired (unpaid after 30 minutes)'
- Return count of expired bookings
- Log: `[INFO] Expired {count} pending bookings`

Also create an API endpoint for triggering expiration (can be called by external cron or internal scheduler):
This will be added as a simple internal endpoint in the route file.
