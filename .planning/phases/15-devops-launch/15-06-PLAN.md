    Create comprehensive beta testing playbook document.

    **Structure:**

    ## 1. Program Overview
    - Goal: Validate ScheduleBox with 3-5 real Czech/Slovak SMBs before public launch
    - Duration: 4 weeks per cohort
    - Success metric: NPS >= 7 average, 2+ testers willing to convert to paid
    - Incentive: 6 months free Growth tier (value 8,940 Kc)

    ## 2. Beta Tester Personas
    Define 3 personas matching target market (per research Pitfall 6):

    | Persona | Profile | Recruitment Channel | Why Important |
    |---------|---------|-------------------|---------------|
    | Beauty Salon Owner | 45-60 years, non-technical, smartphone-primary, 3-8 employees | Local Facebook groups, beauty industry associations | Tests mobile UX, validates onboarding simplicity, biggest market segment |
    | Fitness/Wellness Studio | 30-40 years, moderately technical, uses SaaS tools, 5-15 employees | LinkedIn outreach, fitness networking events | Tests integration capabilities, provides API feedback, validates multi-service booking |
    | Medical/Professional Practice | 50+ years, desktop-first, security-conscious, 2-5 employees | Direct outreach via chambers of commerce, medical associations | Tests compliance features, validates trust signals, critical for premium tier |

    Recruit 3-5 businesses per persona (9-15 total). Minimum viable: 1 per persona (3 total).

    ## 3. Screening Criteria
    - Must have 3+ employees who accept bookings
    - Currently using a booking system (competitor or manual/paper)
    - Willing to commit 4 weeks of active usage
    - Located in Czech Republic or Slovakia
    - Available for 30-min weekly check-in call
    - Willing to sign NDA and provide honest feedback
    - NOT friends/family of the team (avoid selection bias per research)

    ## 4. Onboarding Process
    Week 0 (pre-beta):
    1. Send welcome email with login credentials and quick start guide
    2. Schedule 45-min video onboarding call
    3. During call: set up company profile, import existing customer list (CSV), configure services and employees
    4. Verify: tester can log in, see dashboard, and create a test booking
    5. Add tester to dedicated Slack/WhatsApp support channel

    ## 5. Weekly Feedback Schedule

    **Week 1: Onboarding and Setup**
    - Focus: Account setup, service configuration, employee management
    - Feedback method: 15-min phone call + online survey (5 questions)
    - Questions:
      1. How easy was the initial setup? (1-10)
      2. What was confusing or unclear?
      3. How long did setup take? (minutes)
      4. Would you be able to set this up without help? (yes/no/partially)
      5. What feature did you expect that was missing?

    **Week 2: First Real Bookings**
    - Focus: Customer-facing booking flow, availability management
    - Trigger: After 5th real booking, show in-app NPS survey
    - Feedback method: In-app survey + 15-min call
    - Questions:
      1. How did your customers react to online booking? (positive/neutral/negative)
      2. Did any customer have trouble booking? What happened?
      3. Rate the booking calendar view (1-10)
      4. Did you encounter any bugs? Describe.
      5. NPS: How likely to recommend ScheduleBox to a colleague? (0-10)

    **Week 3: Payment and Notifications**
    - Focus: Payment processing, email/SMS notifications, automation
    - Feedback method: Online survey after first payment + 15-min call
    - Questions:
      1. Did payment processing work correctly? (yes/no, describe issues)
      2. Did your customers receive booking confirmation emails? (yes/no)
      3. Rate the notification templates (1-10)
      4. Did you set up any automation rules? Which ones?
      5. What notification would you want that doesn't exist?

    **Week 4: Full Flow and Retrospective**
    - Focus: Overall experience, missing features, willingness to pay
    - Feedback method: 30-min video retrospective interview
    - Questions:
      1. Overall satisfaction (1-10)
      2. Top 3 things you love
      3. Top 3 things that need improvement
      4. Would you pay 990 Kc/month for this? Why/why not?
      5. What would make you switch from your current system?
      6. Final NPS (0-10)
      7. Would you continue using ScheduleBox after beta? (yes/no)

    ## 6. Success Criteria
    - Average NPS >= 7 across all testers
    - At least 2 testers willing to convert to paid plan
    - No critical bugs reported by Week 4 (all resolved within 48h)
    - Average setup time < 60 minutes with assistance
    - At least 50 real bookings processed across all testers

    ## 7. Support SLA During Beta
    - Bug reports: 4-hour acknowledgment, 24-hour fix for critical, 48h for high
    - Feature requests: Acknowledged within 24h, triaged weekly
    - Support channel: Dedicated Slack/WhatsApp group with dev team access
    - Weekly check-in calls: Mandatory, recorded with permission

    ## 8. Data Collection and Privacy
    - Beta testers sign data processing agreement (GDPR)
    - Anonymize all beta data before internal reporting
    - Testers can export all their data at any time (GDPR export feature)
    - Testers can delete their account after beta ends

    ## 9. Post-Beta Actions
    - Compile feedback report with quantitative metrics and qualitative themes
    - Prioritize top 5 issues for immediate fix
    - Send thank-you email with 6-month free tier activation
    - Schedule 1-month follow-up call to check retention
    - Publish (anonymized) case studies if testers consent

    **docs/deployment-runbook.md:**
    Step-by-step operational runbook for deploying ScheduleBox to Kubernetes.

    ## Prerequisites
    - kubectl configured with cluster access
    - Helm 3.x installed
    - Docker images pushed to ghcr.io
    - DNS records configured for target domain
    - TLS certificate (cert-manager or manual)

    ## Fresh Cluster Setup
    1. Create namespaces: `kubectl create namespace schedulebox-{staging|production}`, `kubectl create namespace monitoring`
    2. Add Helm repos: bitnami, prometheus-community, jaegertracing
    3. Deploy stateful services (in order):
       - PostgreSQL: `helm install postgres bitnami/postgresql --namespace schedulebox-production --values k8s/stateful/postgres-values.yaml`
       - Redis: `helm install redis bitnami/redis --namespace schedulebox-production --values k8s/stateful/redis-values.yaml`
       - RabbitMQ: `helm install rabbitmq bitnami/rabbitmq --namespace schedulebox-production --values k8s/stateful/rabbitmq-values.yaml`
    4. Wait for stateful services: `kubectl wait --for=condition=ready pod -l app.kubernetes.io/name=postgresql --timeout=120s`
    5. Create secrets: `kubectl create secret generic schedulebox-secrets --from-env-file=.env.production`
    6. Run database migrations: `kubectl run migrate --image=ghcr.io/[org]/schedulebox:latest --command -- pnpm db:migrate`
    7. Deploy application: `helm upgrade --install schedulebox ./helm/schedulebox --values helm/schedulebox/values-production.yaml --set web.image.tag=latest`
    8. Deploy monitoring stack: `helm install prometheus prometheus-community/kube-prometheus-stack --namespace monitoring --values k8s/monitoring/prometheus-values.yaml`
    9. Deploy Jaeger: `helm install jaeger jaegertracing/jaeger --namespace monitoring --values k8s/monitoring/jaeger-values.yaml`
    10. Verify: `kubectl get pods -n schedulebox-production` (all Running), `curl https://app.schedulebox.cz/health`

    ## Deploying a New Version
    1. Verify CI passed on the commit
    2. For staging: automatic via deploy-staging.yml workflow on push to main
    3. For production: trigger deploy-production.yml workflow manually with image tag
    4. Monitor rollout: `kubectl rollout status deployment/schedulebox-web -n schedulebox-production`
    5. Verify health: check /health and /readiness endpoints
    6. Check Grafana for error rate spike in first 5 minutes

    ## Rollback Procedures
    1. Quick rollback (Helm): `helm rollback schedulebox 1 --namespace schedulebox-production` (reverts to previous release)
    2. Emergency rollback: `kubectl set image deployment/schedulebox-web web=ghcr.io/[org]/schedulebox:{previous-tag}`
    3. Database rollback: Only if migration was destructive -- `kubectl run migrate-down --image=... --command -- pnpm db:rollback`
    4. Verify after rollback: same health checks as deploy

    ## Scaling
    - Manual: `kubectl scale deployment/schedulebox-web --replicas=5`
    - HPA handles auto-scaling in production (see helm values)
    - To scale AI service: `kubectl scale deployment/schedulebox-ai --replicas=3`

    ## Troubleshooting
    - Pod crash loop: `kubectl logs deployment/schedulebox-web --previous`
    - OOMKilled: increase memory limits in values-production.yaml
    - DNS resolution issues: check CoreDNS pods, enable NodeLocal DNSCache
    - Slow queries: check pg_connections_active metric in Grafana
    - Connection refused to stateful services: verify service endpoints `kubectl get endpoints`
