  <files>
    packages/shared/src/metrics/index.ts

  <action>
    Install `prom-client` in packages/shared: `pnpm --filter @schedulebox/shared add prom-client`

    **packages/shared/src/metrics/index.ts:**
    Create a custom Prometheus registry (do NOT use the default global registry -- avoids conflicts with other libraries).

    Define core metrics:
    - `httpRequestDuration` (Histogram): name `http_request_duration_seconds`, labels `[method, path, status_code]`, buckets `[0.01, 0.05, 0.1, 0.3, 0.5, 0.7, 1, 3, 5, 10]`
    - `httpRequestsTotal` (Counter): name `http_requests_total`, labels `[method, path, status_code]`
    - `dbQueryDuration` (Histogram): name `db_query_duration_seconds`, labels `[query_type]`, buckets `[0.005, 0.01, 0.05, 0.1, 0.3, 0.5, 1, 3]`
    - `pgConnectionsActive` (Gauge): name `pg_connections_active`
    - `redisConnectionsActive` (Gauge): name `redis_connections_active`

    Export: `register`, all metric objects, and `getMetrics()` function that returns `register.metrics()`.

    Also export `collectDefaultMetrics` from prom-client configured with the custom registry and `nodejs_` prefix (provides heap, GC, event loop lag metrics automatically).

    **packages/shared/src/metrics/event-loop.ts:**
    Use `perf_hooks.performance.eventLoopUtilization()` to measure ELU.
    - Create Gauge `nodejs_eventloop_utilization` (0-1 float)
    - `startEventLoopMetrics()` function: starts setInterval(1000ms) that computes delta ELU and sets gauge
    - Export the function (called once at app startup)
    - Guard against multiple invocations (idempotent start)

    **packages/shared/src/metrics/business.ts:**
    Business-specific counters (per project documentation lines 7625-7631):
    - `bookingsTotalCounter` (Counter): `schedulebox_bookings_total`, labels `[company_id, status, source]`
    - `paymentsTotalCounter` (Counter): `schedulebox_payments_total`, labels `[company_id, gateway, status]`
    - `notificationsTotalCounter` (Counter): `schedulebox_notifications_total`, labels `[channel, status]`
    - `aiPredictionDuration` (Histogram): `schedulebox_ai_prediction_duration_seconds`, labels `[model_type]`, buckets `[0.1, 0.5, 1, 2, 5, 10, 15]`

    All registered on the same custom registry from index.ts.

    **apps/web/app/api/metrics/route.ts:**
    - GET handler that calls `getMetrics()` and returns as `text/plain; version=0.0.4; charset=utf-8`
    - This endpoint should be unauthenticated (Prometheus scraper needs direct access)
    - Add a comment noting that in production, this should be restricted via network policy (only monitoring namespace can access)
    - Import and call `startEventLoopMetrics()` at module level (runs once when route module is loaded)
    - Import and call `collectDefaultMetrics()` at module level

    Update `packages/shared/src/index.ts` to re-export from `./metrics/index.js` (add barrel export).

    NOTE: Do NOT integrate metrics middleware into existing route handlers in this plan. That is incremental work for later. This plan establishes the metrics foundation and /api/metrics endpoint.
  </action>

    Run `pnpm type-check` to verify no TypeScript errors.
    Verify /api/metrics route file exists and imports from @schedulebox/shared.
    Check that prom-client is in packages/shared/package.json dependencies.
  </verify>
  <done>
    prom-client installed, 3 metrics modules created (index, event-loop, business), /api/metrics endpoint serves Prometheus text format. All metrics use custom registry with consistent naming.
  </done>

  <action>
    **k8s/monitoring/prometheus-values.yaml:**
    Helm values for `prometheus-community/kube-prometheus-stack`:
    - Prometheus retention: 15d
    - Storage: 50Gi PVC (ReadWriteOnce)
    - additionalScrapeConfigs:
      - job_name `schedulebox-web`: scrape pods with label `app.kubernetes.io/name: schedulebox-web` in production namespace, metrics_path `/api/metrics`, port `http` (3000), scrape_interval 15s
      - job_name `schedulebox-ai`: scrape AI service pods, metrics_path `/metrics`, port 8000
    - Grafana enabled with persistence 10Gi
    - Grafana dashboardProviders configured to load from `/var/lib/grafana/dashboards/default` folder
    - Grafana dashboardsConfigMaps referencing `schedulebox-dashboards` configmap
    - Alertmanager enabled with 10Gi storage
    - Alertmanager config: route grouping by [alertname, cluster, service], group_wait 10s, group_interval 10s, repeat_interval 12h
    - Alertmanager receiver: slack (placeholder webhook URL from values, channel #schedulebox-alerts)
    - Enable nodeExporter, kubeStateMetrics (included by default in stack)

    **k8s/monitoring/grafana-dashboards/schedulebox-overview.json:**
    Create a Grafana dashboard JSON with the following panels (use standard Grafana dashboard JSON format):
    - Row 1 "HTTP Performance":
      - Panel: Request rate (rate(http_requests_total[5m])) -- timeseries
      - Panel: p95 latency (histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))) -- timeseries
      - Panel: Error rate (rate(http_requests_total{status_code=~"5.."}[5m]) / rate(http_requests_total[5m])) -- stat
    - Row 2 "Node.js Runtime":
      - Panel: Event loop utilization (nodejs_eventloop_utilization) -- gauge (0-1, green < 0.5, yellow < 0.7, red > 0.7)
      - Panel: Heap used (nodejs_heap_size_used_bytes) -- timeseries
      - Panel: Active handles (nodejs_active_handles_total) -- timeseries
    - Row 3 "Business Metrics":
      - Panel: Bookings per minute (rate(schedulebox_bookings_total[5m])*60) -- timeseries
      - Panel: Payments by gateway (sum by(gateway) rate(schedulebox_payments_total[5m])) -- pie chart
      - Panel: AI prediction latency (histogram_quantile(0.95, rate(schedulebox_ai_prediction_duration_seconds_bucket[5m]))) -- timeseries
    - Row 4 "Infrastructure":
      - Panel: Pod CPU usage -- timeseries
      - Panel: Pod memory usage -- timeseries
      - Panel: DB connections (pg_connections_active) -- gauge

    Dashboard settings: title "ScheduleBox Overview", uid "schedulebox-overview", auto-refresh 30s, time range last 1h.

    **k8s/monitoring/alerting-rules.yaml:**
    PrometheusRule CRD with alert definitions:
    - `HighErrorRate`: http error rate > 5% for 5 minutes (severity: critical)
    - `HighLatency`: p95 latency > 2s for 5 minutes (severity: warning)
    - `HighLatencyCritical`: p95 latency > 5s for 2 minutes (severity: critical)
    - `EventLoopSaturated`: nodejs_eventloop_utilization > 0.8 for 3 minutes (severity: warning)
    - `PodCrashLooping`: increase(kube_pod_container_status_restarts_total[1h]) > 3 (severity: critical)
    - `DatabaseConnectionsHigh`: pg_connections_active > 8 (80% of 10 pool size) for 2 minutes (severity: warning)
    - All alerts have labels (team: devops, severity) and annotations (summary, description with runbook hints)
  </action>
