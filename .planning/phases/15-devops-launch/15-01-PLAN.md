  <action>
    Create a custom Helm chart (apiVersion: v2) for ScheduleBox with templates for all 3 application services:

    **Chart.yaml:** Name `schedulebox`, version `0.1.0`, appVersion matching current release. No subchart dependencies (stateful services use separate Bitnami charts).

    **_helpers.tpl:** Standard Helm helpers (fullname, labels, selectorLabels, serviceAccountName).

    **values.yaml (defaults):**
    - `global.namespace`: schedulebox
    - `global.imagePullPolicy`: IfNotPresent
    - Per-service blocks: `web`, `aiService`, `notificationWorker`
    - Each service has: `image.repository` (ghcr.io path), `image.tag`, `replicaCount`, `resources.requests` (cpu, memory), `resources.limits`, `env` (list), `service.port`, `service.targetPort`
    - Web defaults: 2 replicas, 250m/256Mi requests, 1000m/512Mi limits, port 3000
    - AI defaults: 1 replica, 500m/512Mi requests, 2000m/1Gi limits, port 8000
    - Worker defaults: 1 replica, 100m/128Mi requests, 500m/256Mi limits
    - Ingress block: enabled, className (nginx), hosts array, tls array
    - HPA block: enabled, minReplicas 2, maxReplicas 10, targetCPU 70, scaleDown stabilization 300s

    **Namespace template:** Creates namespace from `global.namespace` with labels.

    **ConfigMap template:** Contains non-secret env vars (NODE_ENV, LOG_LEVEL, LOG_FORMAT, APP_URL, API_URL, AI_SERVICE_URL as internal K8s DNS).

    **Secrets template:** References for DATABASE_URL, REDIS_URL, RABBITMQ_URL, JWT_SECRET, JWT_REFRESH_SECRET, SENTRY_DSN. Uses `stringData` with values from `{{ .Values.secrets.* }}` placeholders. Add annotation for external-secrets-operator compatibility.

    **web-deployment.yaml:**
    - Container from `ghcr.io/[org]/schedulebox:{{ .Values.web.image.tag }}`
    - Liveness probe: httpGet /health port 3000, initialDelay 30s, period 10s, failureThreshold 3
    - Readiness probe: httpGet /readiness port 3000, initialDelay 5s, period 5s, failureThreshold 2
    - Resource limits from values
    - envFrom: configMapRef + secretRef
    - DNS config: ndots 2 (reduces DNS lookups for external domains -- important for Node.js DNS caching pitfall from research)
    - terminationGracePeriodSeconds: 30

    **web-service.yaml:** ClusterIP service, port 80 -> targetPort 3000

    **web-hpa.yaml:**
    - autoscaling/v2 HPA
    - CPU utilization target 70%
    - Custom metric `nodejs_eventloop_utilization` target 0.6 (from research Pattern 4)
    - Scale-up: immediate (stabilization 0), max 100% increase per 15s
    - Scale-down: 300s stabilization, max 10% decrease per 60s

    **web-ingress.yaml:** Conditionally created when `ingress.enabled`, className from values, hosts/paths/tls from values.

    **ai-deployment.yaml:**
    - Container from `ghcr.io/[org]/schedulebox-ai:{{ .Values.aiService.image.tag }}`
    - Liveness: httpGet /health port 8000, initialDelay 60s (ML model loading), period 15s
    - Readiness: httpGet /health port 8000, initialDelay 30s, period 10s
    - Environment: ENVIRONMENT, AI_SERVICE_PORT, REDIS_URL, SCHEDULEBOX_API_URL (internal K8s DNS), MODEL_DIR
    - Volume mount for model storage (emptyDir or PVC)

    **ai-service.yaml:** ClusterIP, port 8000

    **worker-deployment.yaml:**
    - Same base image as web but command override: `["node", "services/notification-worker/src/index.js"]`
    - No service port (worker, not HTTP)
    - Liveness: exec command `["node", "-e", "process.exit(0)"]` (process alive check)
    - Environment: DATABASE_URL, REDIS_URL, RABBITMQ_URL, SMTP vars, Twilio vars, VAPID vars

    **worker-service.yaml:** Headless service (clusterIP: None) for DNS resolution.

    **values-staging.yaml:** Override replicas (web: 1, ai: 1, worker: 1), smaller resource limits, ingress host `staging.schedulebox.cz`, debug log level.

    **values-production.yaml:** Override replicas (web: 3, ai: 2, worker: 2), full resource limits, ingress host `app.schedulebox.cz`, info log level, HPA enabled.

    **k8s/stateful/postgres-values.yaml:** Bitnami PostgreSQL Helm values -- single primary (no replication for MVP), persistence 50Gi, auth credentials from secret, postgresqlDatabase: schedulebox, resources requests 500m/512Mi.

    **k8s/stateful/redis-values.yaml:** Bitnami Redis values -- standalone (no sentinel for MVP), persistence 10Gi, appendonly yes, maxmemory-policy allkeys-lru, resources 100m/128Mi.

    **k8s/stateful/rabbitmq-values.yaml:** Bitnami RabbitMQ values -- single node, persistence 10Gi, management plugin enabled, resources 250m/256Mi.

    IMPORTANT: Keep Helm charts provider-agnostic (no EKS/GKE-specific annotations). Use standard Kubernetes API objects only. Cloud-specific config goes in values-production.yaml overrides.

  <name>Task 2: GitHub Actions deploy workflows for staging and production</name>
  <files>
    .github/workflows/deploy-staging.yml

  <action>
    Create two GitHub Actions workflows for deploying to Kubernetes via Helm.

    **deploy-staging.yml:**
    - Trigger: push to `main` branch (after CI passes)
    - Jobs: `deploy` (runs-on: ubuntu-latest)
    - Steps:
      1. Checkout code
      2. Setup Helm (azure/setup-helm@v4)
      3. Configure kubectl (azure/k8s-set-context@v4 with kubeconfig from secrets.KUBE_CONFIG_STAGING)
      4. Add Bitnami Helm repo, update
      5. Deploy stateful services (helm upgrade --install for postgres, redis, rabbitmq using k8s/stateful/ values with --namespace schedulebox-staging --create-namespace)
      6. Deploy app (helm upgrade --install schedulebox ./helm/schedulebox --namespace schedulebox-staging --values helm/schedulebox/values-staging.yaml --set web.image.tag=${{ github.sha }} --set aiService.image.tag=${{ github.sha }} --wait --timeout 5m)
      7. Verify deployment: kubectl rollout status deployment/schedulebox-web -n schedulebox-staging --timeout=120s
    - Environment: staging (with required reviewers: none for auto-deploy)
    - Secrets needed: KUBE_CONFIG_STAGING

    **deploy-production.yml:**
    - Trigger: workflow_dispatch (manual) with input `image_tag` (string, required, default: latest)
    - Also trigger on release published
    - Jobs: `deploy` (runs-on: ubuntu-latest, environment: production)
    - Steps: Same as staging but:
      1. Uses secrets.KUBE_CONFIG_PRODUCTION
      2. Uses values-production.yaml
      3. Namespace: schedulebox-production
      4. Image tag from input or github.sha
      5. Adds `--atomic` flag for automatic rollback on failure
    - Environment: production (with required reviewers -- GitHub settings)
    - Secrets needed: KUBE_CONFIG_PRODUCTION

    Both workflows should:
    - Reference existing CI workflow as dependency (needs: [] but document that CI should pass first)
    - Use concurrency group to prevent parallel deploys to same environment
    - Include step to output deployed version for audit trail

    NOTE: These workflows will not run successfully until the user configures the K8s cluster and adds KUBE_CONFIG secrets. That is expected -- the code artifacts are what we produce here.
  </action>

    Verify Helm commands reference correct values files and namespaces.
  </verify>
