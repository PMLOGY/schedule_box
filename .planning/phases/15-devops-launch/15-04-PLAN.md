  <action>
    Create the load test directory structure with 4 k6 test scenarios.

    **load-tests/helpers/auth.js:**
    Helper module for authenticating test users:
    - `login(baseUrl, email, password)` function: POST /api/v1/auth/login, returns { accessToken, refreshToken }
    - `getAuthHeaders(token)` function: returns { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' }
    - Handle login failure gracefully (log error, return null)

    **load-tests/fixtures/users.json:**
    Array of 50 test user objects with: email, password, name, phone. Use pattern `loadtest-{N}@schedulebox.cz` for emails. All passwords `LoadTest123!@#`. These are seeded into the test database before load testing.

    **load-tests/scenarios/booking-flow.js:**
    Full booking lifecycle test (60% of expected traffic):
    - Stages: ramp to 100 VUs (2m), ramp to 600 VUs (5m), sustain 600 VUs (10m), ramp down (2m)
    - Thresholds: `http_req_duration: ['p(95)<2000', 'p(99)<5000']`, `http_req_failed: ['rate<0.01']`
    - Flow per iteration:
      1. Login (use SharedArray of users, round-robin by VU)
      2. GET /api/v1/services (list services)
      3. GET /api/v1/availability?serviceId={id}&date={tomorrow} (check slots)
      4. POST /api/v1/bookings (create booking with available slot)
      5. GET /api/v1/bookings/{id} (verify booking created)
    - Add `check()` assertions on each response (status codes, body shape)
    - Sleep 1-3s between requests (realistic user think time)
    - Use `__ENV.BASE_URL` for target URL (injected from environment)

    **load-tests/scenarios/availability-lookup.js:**
    Read-heavy availability checking (30% of expected traffic):
    - Stages: ramp to 100 VUs (1m), ramp to 300 VUs (3m), sustain 300 VUs (10m), ramp down (1m)
    - Thresholds: `http_req_duration: ['p(95)<1000', 'p(99)<3000']`, `http_req_failed: ['rate<0.01']`
    - Flow per iteration:
      1. GET /api/v1/availability for random service, date range (today + 0..6 days)
      2. GET /api/v1/services/{id} (service detail)
    - This is the most latency-sensitive endpoint -- tighter thresholds
    - Sleep 0.5-1.5s between requests

    **load-tests/scenarios/api-mixed.js:**
    Combined traffic pattern matching real-world distribution:
    - Uses k6 scenarios (not stages) for weighted traffic:
      - `booking_users` scenario: 60% weight, executor `ramping-vus`, max 600 VUs
      - `browsing_users` scenario: 30% weight, executor `ramping-vus`, max 300 VUs
      - `admin_users` scenario: 10% weight, executor `ramping-vus`, max 100 VUs
    - Total target: ramp to 1000 VUs sustained for 10 minutes
    - booking_users: run booking-flow logic
    - browsing_users: run availability-lookup logic
    - admin_users: GET /api/v1/bookings (list), GET /api/v1/customers (list), GET /api/v1/analytics/overview
    - Thresholds per scenario group using tags: `http_req_duration{scenario:booking_users}: ['p(95)<2000']`
    - Overall thresholds: p95 < 2s, p99 < 5s, error rate < 1%

    **load-tests/scenarios/spike-test.js:**
    Sudden traffic surge test:
    - Stages: baseline 100 VUs (3m), spike to 2000 VUs (30s), sustain spike 1m, drop to 100 (30s), recover 3m
    - Thresholds: looser during spike -- `http_req_duration: ['p(95)<5000']`, `http_req_failed: ['rate<0.05']` (5% acceptable during burst)
    - Same booking flow logic but validates system recovery (no cascading failures)
    - Custom counter for tracking requests during spike vs baseline periods

    **load-tests/README.md:**
    Document:
    - Prerequisites (k6 installed, test database seeded, staging URL accessible)
    - How to run each scenario: `k6 run --env BASE_URL=https://staging.schedulebox.cz scenarios/booking-flow.js`
    - How to interpret results (explain p95, p99, VUs, checks, thresholds)
    - Test data seeding instructions (reference seed script or manual SQL)
    - Expected resource requirements: 1000 VUs needs 4+ CPU cores, 8GB RAM on test machine
    - Threshold meanings: p95 < 2s (good UX), p99 < 5s (no outlier horror), error < 1% (reliable)

  <action>
    Create GitHub Actions workflow for running k6 load tests.

    **Workflow triggers:**
    - `workflow_dispatch` with inputs:
      - `scenario` (choice: booking-flow, availability-lookup, api-mixed, spike-test, default: api-mixed)
      - `base_url` (string, required, default: https://staging.schedulebox.cz)
      - `duration_multiplier` (string, default: '1', allows scaling test duration e.g., '0.5' for half-length)
    - `schedule`: weekly Sunday 3am UTC (cron: '0 3 * * 0') for regression testing

    **Job: load-test**
    - runs-on: ubuntu-latest (4 CPU, 16GB RAM -- sufficient for up to ~1000 VUs)
    - Steps:
      1. Checkout code
      2. Install k6 (use grafana/setup-k6-action@v1 or curl install)
      3. Run k6 with selected scenario:
         ```
         k6 run --env BASE_URL=${{ inputs.base_url || 'https://staging.schedulebox.cz' }} \
           --summary-export=summary.json \
           --out json=results.json \
           load-tests/scenarios/${{ inputs.scenario || 'api-mixed' }}.js
         ```
      4. Upload test results as artifact (summary.json, results.json)
      5. Post summary to PR comment (if triggered from PR) using actions/github-script
      6. Fail workflow if k6 exits with non-zero (thresholds breached)

    **Job: report** (needs: load-test, if: always())
    - Parse summary.json and create markdown report with:
      - Pass/fail status per threshold
      - p50, p95, p99 latencies
      - Total requests, error rate
      - VU peak count
    - Upload as workflow summary (using `$GITHUB_STEP_SUMMARY`)

    Add comments noting:
    - For 1000+ VU tests, consider running k6 from a dedicated machine (not CI runner)
    - k6 Cloud or distributed k6 (k6 Operator) needed for >1000 VU tests from CI
  </action>
