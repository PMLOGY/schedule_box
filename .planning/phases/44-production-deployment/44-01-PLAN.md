**1. Create `docker/docker-compose.prod.yml`:**

Production-only compose file with three services:

- **postgres**: Same as dev (postgres:16-alpine) but with `env_file: ../.env` for POSTGRES_PASSWORD. Map environment vars: POSTGRES_DB=schedulebox, POSTGRES_USER=schedulebox, POSTGRES_PASSWORD from env. Volume: postgres_data. Healthcheck same as dev. Port 5432 exposed only internally (no host mapping — app connects via Docker network). Restart: unless-stopped.

- **redis**: Same as dev (redis:7-alpine). Volume: redis_data. Healthcheck same as dev. Command: `redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru`. Port 6379 internal only. Restart: unless-stopped.

- **app**: Build from `context: ..`, `dockerfile: docker/Dockerfile`, `target: production`. No volume mounts (standalone build baked into image). Depends on postgres (service_healthy) and redis (service_healthy). Port mapping: `${PORT:-3000}:3000`. env_file: `../.env`. Key environment overrides (override .env for Docker networking): `DATABASE_URL=postgresql://schedulebox:${POSTGRES_PASSWORD}@postgres:5432/schedulebox`, `REDIS_URL=redis://redis:6379`, `NODE_ENV=production`. Restart: unless-stopped. Add an init container or entrypoint command that runs migrations before starting. Use this approach: add a `command` override that runs migrations then starts the app: `sh -c "node packages/database/src/migrate.js || true && node apps/web/server.js"`. Actually, since the production stage only has the standalone output and no pnpm/migration tooling, instead create a separate one-off `migrate` service:

  ```yaml
  migrate:
    build:
      context: ..
  ```

**2. Create `apps/web/lib/env.ts`:**

Use Zod to define and validate environment variables. Split into required and optional groups:

Required (app crashes without these):
- `DATABASE_URL` (z.string().url() or z.string().startsWith('postgresql://'))
- `REDIS_URL` (z.string())
- `JWT_ACCESS_SECRET` (z.string().min(16) in production, any in dev)
- `JWT_REFRESH_SECRET` (z.string().min(16) in production, any in dev)
- `NEXT_PUBLIC_APP_URL` (z.string().url())

Optional with defaults:
- `NODE_ENV` (z.enum(['development', 'production', 'test']).default('development'))
- `PORT` (z.coerce.number().default(3000))
- `JWT_ACCESS_EXPIRY` (z.string().default('15m'))
- `JWT_REFRESH_EXPIRY` (z.string().default('7d'))
- `TZ` (z.string().default('Europe/Prague'))
- `AI_SERVICE_URL` (z.string().optional())
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` (all z.string().optional())
- `COMGATE_MERCHANT_ID`, `COMGATE_SECRET` (z.string().optional())
- `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_FROM_NUMBER` (z.string().optional())
- `CRON_SECRET` (z.string().optional())

Export a `validateEnv()` function that:
1. Parses `process.env` against the Zod schema
2. On success: logs "Environment validated successfully" and returns parsed env
3. On failure: logs each missing/invalid variable clearly (variable name + error message), then throws an Error with summary "Missing or invalid environment variables: VAR1, VAR2"

Also export the parsed `env` object for type-safe access elsewhere (lazy-initialized on first access).

**3. Create `apps/web/instrumentation.ts`:**

Next.js instrumentation file (runs once on server startup). Content:

```typescript
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    const { validateEnv } = await import('./lib/env');
    validateEnv();
  }
}
```

This uses dynamic import so the validation only runs server-side, not in Edge runtime.
