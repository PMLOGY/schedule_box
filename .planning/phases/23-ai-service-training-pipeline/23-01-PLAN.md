  <action>
    Create `apps/web/lib/middleware/ai-service-auth.ts`:
    - Export `validateAiServiceKey(req: NextRequest): NextResponse | null`
    - Read `x-ai-service-key` header from the request
    - Compare against `process.env.AI_SERVICE_API_KEY`
    - If `AI_SERVICE_API_KEY` env var is not set (empty/undefined), skip auth (development mode) by returning `null`
    - If header is missing or doesn't match, return `NextResponse.json({ error: 'Unauthorized', code: 'INVALID_API_KEY' }, { status: 401 })`
    - If auth passes, return `null`
    - Do NOT use the existing `createRouteHandler` pattern — these are internal machine-to-machine routes that don't use JWT, RBAC, or rate limiting. Use simple inline auth via `validateAiServiceKey()` at the top of each route handler.

    Create `apps/web/app/api/internal/features/training/no-show/route.ts`:
    - Export async `GET` handler
    - Call `validateAiServiceKey(req)` first; return error response if non-null
    - Import `db` from `@schedulebox/database` and `sql` from `drizzle-orm`
    - Execute raw SQL query that returns feature rows matching `NoShowPredictor.FEATURE_COLUMNS`:
      ```
      booking_lead_time_hours, customer_no_show_rate, customer_total_bookings,
      day_of_week, hour_of_day, is_weekend, service_duration_minutes,
      service_price, is_first_visit, has_payment, days_since_last_visit, no_show (target)
      ```
    - Query joins bookings (status IN 'completed', 'no_show') with services for price/duration
    - Uses LATERAL subqueries for customer stats (no_show_rate, total_bookings) and days_since_last_visit
    - LIMIT 5000 rows
    - See RESEARCH.md Pattern 1 for the exact SQL query
    - Return `NextResponse.json(rows)`

    Create `apps/web/app/api/internal/features/training/clv/route.ts`:
    - Export async `GET` handler with same auth pattern
    - Query customers table joined with booking aggregates to produce CLV feature rows:
      ```
      total_bookings, total_spent, avg_booking_value, days_since_first_visit,
      days_since_last_visit, booking_frequency, no_show_rate, service_diversity, future_clv (target)
      ```
    - `future_clv` = total_spent * (1 + ln(booking_frequency + 1)) * (1 - no_show_rate) as proxy target
    - Filter customers with at least 2 bookings (need history for meaningful CLV)
    - LIMIT 5000 rows

    Create `apps/web/app/api/internal/features/training/capacity/route.ts`:
    - Export async `GET` handler with same auth pattern
    - Query bookings grouped by hour (date_trunc('hour', start_time)) to produce Prophet-compatible data:
      ```
      ds (datetime), y (booking count per hour)
      ```
    - Filter bookings with status IN ('completed', 'confirmed', 'no_show') — all real bookings
    - Order by ds ascending
    - Return as JSON array of {ds, y} objects
  </action>

    - `ai-service-auth.ts` exports `validateAiServiceKey`

    Auth middleware created; no-show, CLV, and capacity training routes return feature rows from the database with API key validation

    Create `apps/web/app/api/internal/features/training/upselling/route.ts`:
    - Export async `GET` handler with `validateAiServiceKey` auth
    - Query to build a service co-booking matrix: for each pair of services booked by the same customer, return a row with `service_a_id`, `service_b_id`, `co_booking_count`
    - SQL: self-join bookings on customer_id where service_id differs, group by service pair, count occurrences
    - Filter to completed bookings only
    - LIMIT 10000 rows

    Create `apps/web/app/api/internal/features/training/reminder-timing/route.ts`:
    - Export async `GET` handler with `validateAiServiceKey` auth
    - Query notification logs + bookings to produce reminder timing training data:
      ```
      customer_id, notification_channel, minutes_before_booking, was_opened (boolean), booking_status
      ```
    - Join notifications with bookings on booking reference
    - Filter to reminder-type notifications that have been sent
    - LIMIT 5000 rows

    Create `apps/web/app/api/internal/features/training/pricing/route.ts`:
    - Export async `GET` handler with `validateAiServiceKey` auth
    - Query booking/pricing outcome data:
      ```
      service_id, price_charged, day_of_week, hour_of_day, booking_status, utilization_estimate
      ```
    - `utilization_estimate` = count of bookings in same hour / estimated capacity (use 5 as default capacity)
    - Filter to completed and no_show bookings
    - LIMIT 5000 rows

    All three routes follow the same pattern: import validateAiServiceKey, check auth at top of handler, use db.execute(sql`...`), return NextResponse.json(rows).
