  <name>Task 1: Configure MSW 2.0 with handlers for Comgate, AI service, and SMTP</name>
  <files>

  <action>
1. Install MSW in apps/web:
   `pnpm add -D --filter @schedulebox/web msw`

2. Create `apps/web/mocks/handlers.ts` with default MSW request handlers for external APIs:

   **Comgate Payment Gateway:**
   - `http.post('https://payments.comgate.cz/v1.0/create', ...)` -> return JSON `{ code: 0, message: 'OK', transId: 'TEST-12345', redirect: 'https://payments.comgate.cz/client/instructions/index?id=TEST-12345' }`
   - `http.post('https://payments.comgate.cz/v1.0/status', ...)` -> return JSON `{ code: 0, message: 'OK', status: 'PAID', amount: 500 }`
   - `http.post('https://payments.comgate.cz/v1.0/refund', ...)` -> return JSON `{ code: 0, message: 'OK' }`

   **AI Service (Python microservice):**
   - `http.post('http://localhost:8000/api/v1/predict/no-show', ...)` -> return JSON `{ probability: 0.15, factors: ['new_customer', 'weekend'] }`
   - `http.post('http://localhost:8000/api/v1/predict/demand', ...)` -> return JSON `{ predictions: [{ date: '2026-02-20', demand: 42 }, { date: '2026-02-21', demand: 38 }] }`
   - `http.post('http://localhost:8000/api/v1/optimize/pricing', ...)` -> return JSON `{ suggestedPrice: 550, confidence: 0.82 }`

   **SMTP / Notification endpoints (if app calls them internally):**
   - If the notification worker is called via HTTP, mock those endpoints
   - Otherwise, export a helper `createSMTPHandler(status: number)` that creates handlers returning success/failure for SMTP-like calls
   - At minimum, provide a handler for any internal notification API: `http.post('/api/v1/notifications/send', ...)` -> return JSON `{ success: true, messageId: 'test-msg-123' }`

   Export all handlers as a `handlers` array. Also export individual handler groups (`comgateHandlers`, `aiHandlers`, `notificationHandlers`) so tests can cherry-pick.

   Use MSW 2.0 API: `import { http, HttpResponse } from 'msw'`. NOT the deprecated `rest` API.

3. Create `apps/web/mocks/server.ts`:
   ```typescript
   import { setupServer } from 'msw/node'
   import { handlers } from './handlers'
   export const server = setupServer(...handlers)
   ```

4. Create `apps/web/vitest.setup.ts`:
   - Import `beforeAll`, `afterEach`, `afterAll` from 'vitest'
   - Import `server` from './mocks/server'
   - `beforeAll(() => server.listen({ onUnhandledRequest: 'warn' }))` -- warn on unmocked requests (don't error, too strict for initial setup)
   - `afterEach(() => server.resetHandlers())` -- CRITICAL: reset between tests
   - `afterAll(() => server.close())`
   - Optionally import `@testing-library/jest-dom/vitest` for DOM matchers (toBeInTheDocument, etc.)

5. Update `apps/web/vitest.config.ts`:
   - Add `setupFiles: ['./vitest.setup.ts']` to the test config
   - Ensure the setup file path is correct relative to apps/web/

6. Create a verification test `apps/web/__tests__/msw-handlers.test.ts`:
   - Import `server` from '../mocks/server'
   - Import `http, HttpResponse` from 'msw'
   - Test that default Comgate handler returns expected response:
     ```typescript
     test('Comgate create payment handler returns success', async () => {
       const response = await fetch('https://payments.comgate.cz/v1.0/create', {
         method: 'POST',
         body: new URLSearchParams({ merchant: 'test', amount: '500' }),
       });
       const data = await response.json();
       expect(data.code).toBe(0);
       expect(data.transId).toBe('TEST-12345');
     });
     ```
   - Test handler override:
     ```typescript
     test('can override handler for error scenario', async () => {
       server.use(
         http.post('https://payments.comgate.cz/v1.0/create', () => {
           return HttpResponse.json({ code: 1, message: 'Merchant not found' }, { status: 400 });
         })
       );
       const response = await fetch('https://payments.comgate.cz/v1.0/create', { method: 'POST' });
       const data = await response.json();
       expect(data.code).toBe(1);
     });
     ```
  </action>

<task type="auto">
  <name>Task 2: Add unit test job to CI pipeline with coverage enforcement</name>
  <files>
    .github/workflows/ci.yml
    package.json
  </files>
  <action>
1. Read the existing `.github/workflows/ci.yml` and update it to add a test job.

2. Add a new job `test` between the existing `lint` and `build` jobs:

```yaml
  test:
    name: Unit Tests
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Run unit tests with coverage
        run: pnpm test:coverage

      - name: Upload coverage report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: |
            packages/shared/coverage/
            packages/events/coverage/
            apps/web/coverage/
          retention-days: 14
```

3. Update the `build` job's `needs` to include `test`:
   Change `needs: lint` to `needs: [lint, test]` on the build job.
   This ensures the build only runs if both lint AND tests pass.

4. Ensure the `test:coverage` script in root package.json runs Vitest with coverage and fails on threshold violations. The script should be: `"test:coverage": "vitest run --coverage"`. Vitest automatically fails if coverage drops below thresholds defined in vitest.shared.ts.

5. IMPORTANT: Do NOT add a separate coverage check step. Vitest's built-in `thresholds` config (lines: 80, functions: 80, etc.) already causes a non-zero exit code when thresholds are not met. This is configured in vitest.shared.ts (Plan 01).

NOTE: The coverage thresholds should NOT fail the initial CI run since we only have a few test files. To handle the bootstrap phase, consider one of:
- Option A: Set `thresholds.perFile: false` (default) so overall project coverage is checked, not per-file
- Option B: Add `thresholds.autoUpdate: false` (don't auto-update, keep 80% as target)
- The thresholds will naturally be met once Plan 02 adds the unit tests

If the coverage thresholds would block the CI initially (before Plan 02 tests exist), add a note in the test job: the thresholds will be fully enforced after Plan 02 completes. For now, run tests without `--coverage.thresholds` failing the build by temporarily using `pnpm test` (not `pnpm test:coverage`) in CI. Then update to `pnpm test:coverage` after Plan 02.

ACTUALLY: Since Plan 02 and 03 are in the same wave, Plan 02 will add the tests. The CI workflow will be committed to git and only takes effect on push. So by the time the developer pushes, all tests from Plan 02 exist. Use `pnpm test:coverage` directly.
