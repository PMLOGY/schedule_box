<objective>
Fix the critical webhook signature verification mismatch so Comgate payment callbacks work in production.

Purpose: The current webhook handler uses HMAC-SHA256 header verification, but Comgate actually sends the merchant secret as a POST body parameter (evidence from PHP SDK, Node SDK, Clojure client). Without this fix, ALL production webhooks would return 401 and no payment would ever be confirmed.

Output: Updated webhook handler, updated client exports, updated integration tests.
</objective>

  <files>apps/web/app/api/v1/payments/comgate/client.ts, apps/web/app/api/v1/webhooks/comgate/route.ts</files>
  <action>
In `apps/web/app/api/v1/payments/comgate/client.ts`:
1. Remove the `verifyComgateSignature` function entirely (lines 241-270). It uses HMAC-SHA256 header verification which does not match Comgate's actual API behavior.
2. Add a new exported function `verifyComgateWebhookSecret(receivedSecret: string): boolean` that:
   - Calls `getComgateCredentials()` to get the expected secret
   - Uses `crypto.timingSafeEqual` to compare `receivedSecret` with the expected secret (constant-time comparison to prevent timing attacks)
   - Returns `true` if they match, `false` otherwise
   - Handles length mismatch gracefully (return false if lengths differ, per existing pattern)
3. Remove the `import crypto from 'crypto'` only if no other function uses it. Keep it since `verifyComgateWebhookSecret` still needs `crypto.timingSafeEqual`.

In `apps/web/app/api/v1/webhooks/comgate/route.ts`:
1. Replace `import { verifyComgateSignature }` with `import { verifyComgateWebhookSecret, getComgatePaymentStatus }` from the client.
2. Remove the header-based signature check (lines 44-51 reading `x-signature`/`signature` headers).
3. After parsing the body with `new URLSearchParams(rawBody)`, extract `parsedBody.get('secret')` and verify using `verifyComgateWebhookSecret(receivedSecret)`. If verification fails, return 401 with `{ error: 'Invalid secret' }`.
4. After finding the payment by gateway transaction ID (step 7), add a defense-in-depth API status check: call `getComgatePaymentStatus(transId)` and compare `apiStatus.status` with the webhook `status` parameter. If they disagree, log a warning but trust the API response (it's authoritative). Use the API-verified status for the subsequent PAID/CANCELLED/AUTHORIZED branching.
5. Keep all other webhook logic unchanged (idempotency, SAGA handlers, event publishing, error handling).

Important: The `secret` field in the POST body is the COMGATE_SECRET env var value that Comgate echoes back. This is NOT a signature -- it's a shared secret comparison.
