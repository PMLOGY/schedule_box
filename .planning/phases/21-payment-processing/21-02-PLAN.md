<objective>
Create a cron-safe payment expiration endpoint and fix environment variable documentation to match actual code.

Purpose: The existing expire-pending endpoint requires user authentication (SETTINGS_MANAGE permission), making it unusable by an external cron service (Railway Cron, cron-job.org). A separate cron endpoint with bearer token auth enables automated payment timeout without human intervention. The env var docs also have discrepancies (COMGATE_API_KEY vs COMGATE_SECRET, COMGATE_TEST_MODE vs NODE_ENV toggle) that would confuse production setup.

Output: New cron endpoint, corrected documentation, updated .env.example.
</objective>

2. Authentication: Check `Authorization: Bearer <token>` header against `process.env.CRON_SECRET`. If CRON_SECRET is not set, return 503 with `{ error: 'Cron endpoint not configured' }`. 4. Call `expirePendingPayments()` (no arguments -- uses default 30min or PAYMENT_TIMEOUT_MINUTES env var).
5. Return 200 with `{ expired_count: N, timestamp: new Date().toISOString() }`.
6. Wrap in try/catch: on error, log and return 500 with `{ error: 'Payment expiration failed' }`.
7. Do NOT use `createRouteHandler` -- this is a simple standalone route with no auth middleware, no Zod schema, no tenant scoping. Keep it minimal.
8. Add a comment at the top: "Called by external cron service (Railway Cron / cron-job.org) every 5-10 minutes."

Add `CRON_SECRET` to `.env.example`:


The existing authenticated endpoint at `/api/v1/payments/expire-pending` remains unchanged -- it's still useful for manual admin triggering.
