<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Configure Comgate production credentials and Railway environment</name>
  <action>
This task requires human action because Comgate KYC verification and Railway dashboard configuration cannot be automated by Claude.

**Comgate Setup (portal.comgate.cz):**
1. Register/login at comgate.cz
2. Complete KYC verification (upload business documents -- may take days/weeks)
3. Once approved, note your production Merchant ID and Secret from portal.comgate.cz
4. In portal settings, set the webhook notification URL to: `https://app.schedulebox.cz/api/v1/webhooks/comgate`
5. Whitelist your Railway server's outbound IP address in the Allowed IPs section
   - To find Railway's IP: check Railway docs for static outbound IPs, or deploy a simple endpoint that logs the request source IP

**Railway Setup (railway.app dashboard):**
1. Add environment variables to the ScheduleBox web service:
   - `COMGATE_MERCHANT_ID` = your production merchant ID
   - `COMGATE_SECRET` = your production secret key
   - `CRON_SECRET` = generate with `openssl rand -hex 32`
2. Verify existing variables are correct:
   - `NEXT_PUBLIC_APP_URL` = `https://app.schedulebox.cz` (NOT localhost)
   - `NODE_ENV` = `production`
   - `COMGATE_API_URL` = `https://payments.comgate.cz` (or omit for default)
3. Set up cron job for payment expiration:
   - Create a Railway Cron Service (or use cron-job.org)
   - Schedule: every 5 minutes (`*/5 * * * *`)
   - URL: `POST https://app.schedulebox.cz/api/v1/payments/expire-pending/cron`
   - Header: `Authorization: Bearer <your-CRON_SECRET-value>`
4. Redeploy the web service to pick up new environment variables
  </action>

  <how-to-verify>
**Test 1: Payment Creation**
1. Log in to https://app.schedulebox.cz as a business owner
2. Create a test booking for a paid service
3. Initiate payment -- you should be redirected to Comgate's payment page
4. Use Comgate's test card (4000000000000002) or a real card for a small amount
5. Complete the payment
6. Expected: You are redirected back to the app, booking status changes to "confirmed"

**Test 2: Webhook Verification**
1. Check Railway logs for the webhook processing:
   - Should see "Webhook processed" (not "Invalid secret")
   - Should see payment status update logged
2. If webhook fails with 401, the secret verification needs debugging

**Test 3: Refund**
1. Find the test payment in the admin panel
2. Initiate a full refund
3. Expected: Payment status changes to "refunded", money returns to card (may take 1-5 business days)

**Test 4: Cron Expiration**
1. Create a new booking with payment but do NOT complete the payment
2. Wait 30+ minutes (or temporarily set PAYMENT_TIMEOUT_MINUTES=1 in Railway for faster testing)
3. Verify the cron endpoint has been called (check Railway cron logs)
4. Expected: Payment status changes to "failed" with reason "payment_timeout", booking cancels

**Test 5: Error Handling**
1. Try to pay for an already-confirmed booking -- should return error
2. Try to refund an unpaid booking -- should return error
