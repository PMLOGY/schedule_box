<task type="auto">
  <name>Task 1: Verify public booking wizard end-to-end</name>
  <files>apps/web/app/[locale]/[company_slug]/**, apps/web/app/api/v1/bookings/route.ts, apps/web/app/api/v1/bookings/availability/route.ts, apps/web/lib/payment/**</files>
  <action>
1. Identify the public booking URL pattern. Check the [company_slug] route structure:
   - List files in apps/web/app/[locale]/[company_slug]/ to understand the wizard steps
   - Determine the seed company's slug (check the database seed file or query: SELECT slug FROM companies LIMIT 5)

2. Test the public booking page loads:
   ```
   curl -s http://localhost:3000/cs/{company-slug} -o /dev/null -w "%{http_code}"
   ```
   If 404: Check the dynamic route parameter name matches the DB column. If 500: Read error, fix SSR crash.

3. Test service listing on the public page:
   ```
   curl -s http://localhost:3000/api/v1/bookings/services?companySlug={slug}
   ```
   Or however the public booking fetches services. The endpoint must return services without requiring auth.
   Fix: Missing public (no-auth) route variant, wrong company lookup, empty results.

4. Test availability/slot fetching:
   ```
   curl -s "http://localhost:3000/api/v1/bookings/availability?serviceId={id}&date=2026-03-28"
   ```
   Must return time slots. Fix: availability engine errors, missing working hours, timezone issues.

5. Test booking creation (the core action):
   ```
   curl -X POST http://localhost:3000/api/v1/bookings \
     -H "Content-Type: application/json" \
     -d '{"serviceId":"...","employeeId":"...","date":"2026-03-28","time":"10:00","customerName":"Test Customer","customerEmail":"test@customer.com","customerPhone":"+420123456789"}'
   ```
   Expected: 201 with booking object. Fix any issues:
   - Validation failures (Zod schema mismatch)
   - Slot already taken race condition
   - Missing customer record creation
   - Payment initiation failure (Comgate API call)

6. If the booking flow involves Comgate payment:
   - Check if payment initiation works or gracefully handles missing Comgate credentials
   - For businesses without payment config, booking should still work (free booking or pay-on-site)
   - Verify the payment redirect URL is correct (should point to the company's booking confirmation, not a broken URL)

7. Verify the created booking appears in the owner's dashboard:
   ```
   curl -s http://localhost:3000/api/v1/bookings -b cookies.txt
   ```
   (Using the owner's auth cookie from Plan 02). The new booking should be in the list.

8. Common bugs:
   - Public page tries to read auth context and crashes for unauthenticated users
   - Availability engine returns no slots (working hours not seeded, timezone offset wrong)
   - Booking creation requires auth but public wizard has no auth
   - Payment callback URL uses Vercel domain instead of localhost/Coolify domain
   - Company slug lookup is case-sensitive but slugs are stored lowercase
  </action>
  <verify>
    <automated>curl -s http://localhost:3000/api/health -w "%{http_code}" | tail -c 3</automated>
  </verify>
  <done>Public booking page loads for a known company slug. Services and available slots display correctly. A booking can be created via the API. The booking appears in the owner's booking list. Payment initiation either works or gracefully degrades when credentials are missing.</done>
</task>
