**Update next.config.mjs:**
Add image optimization configuration:
```javascript
images: {
  formats: ['image/avif', 'image/webp'],
  deviceSizes: [640, 750, 828, 1080, 1200, 1920],
  imageSizes: [16, 32, 48, 64, 96, 128, 256],
},
```

Add `@react-pdf/renderer` to `serverExternalPackages` array (alongside existing `pdfkit`). This prevents bundling issues with the PDF library in the edge runtime.

Add `modularizeImports` for lucide-react to tree-shake unused icons and reduce bundle size:
```javascript
modularizeImports: {
  'lucide-react': {
    transform: 'lucide-react/dist/esm/icons/{{kebabCase member}}',
  },
},
```

Note: If modularizeImports causes issues with lucide-react's barrel exports, remove it and rely on Next.js built-in tree-shaking instead.

**Update root layout.tsx:**
The Inter font is already imported with `next/font/google`. Verify it uses `subsets: ['latin', 'latin-ext']` (it does). Add `display: 'swap'` to the Inter font config if not already present -- this ensures text remains visible during font load (FOUT over FOIT for better LCP). Add `preload: true` explicitly (this is the default but making it explicit documents the intent).

Also add the font variable to the html class for Tailwind integration if not already done:
```tsx
const inter = Inter({
  subsets: ['latin', 'latin-ext'],
  display: 'swap',
  variable: '--font-inter',
});
// Then on <body>: className={`${inter.variable} font-sans`}
```

Verify the existing Inter setup and only modify if `display: 'swap'` is missing. Do NOT break the existing font rendering.

**Update BookingCalendar.tsx:**
The FullCalendar component is a large library (~200KB). If it's not already dynamically imported, wrap the FullCalendar import with:
```typescript
const FullCalendar = dynamic(() => import('@fullcalendar/react'), {
  ssr: false,
  loading: () => <div className="h-[600px] animate-pulse bg-muted rounded-lg" />,
});
```

Check if the calendar page already uses dynamic import. If the component itself is a `'use client'` component imported directly, the dynamic import should be at the PAGE level where the component is used, not inside the component. Apply the dynamic import in the correct location (likely `apps/web/app/[locale]/(dashboard)/calendar/page.tsx`).

Also check `apps/web/app/[locale]/(dashboard)/automation/builder/page.tsx` for React Flow imports. If `@xyflow/react` is imported directly (not dynamically), wrap it with `dynamic(() => import(...), { ssr: false })` at the page level.

The goal is: all components using browser-only APIs or large bundles (FullCalendar, @xyflow/react, Recharts) use dynamic imports with ssr: false.

  <action>
Conduct a systematic accessibility audit of existing components and apply fixes. This is NOT about running axe-core (that requires a running browser). This is about reading component source code and applying known WCAG 2.1 AA patterns.

**Audit and fix sidebar.tsx:**
- Add `aria-label="Main navigation"` to the `<nav>` element inside the sidebar.
- Add `aria-current="page"` to the active nav link (when `isActive` is true).
- The collapse/expand button needs `aria-label`: `aria-label={sidebarCollapsed ? 'Expand sidebar' : 'Collapse sidebar'}` (or use translations).
- Add `aria-expanded={!sidebarCollapsed}` to the toggle button.

**Audit and fix header.tsx:**
- Verify the mobile menu button already has `aria-label` from Plan 02 (if not, add it).
- Add `aria-label="User menu"` to the UserMenu trigger if it uses a DropdownMenuTrigger.
- If the Breadcrumbs component renders a `<nav>`, ensure it has `aria-label="Breadcrumb"`.

**Audit and fix dashboard layout.tsx:**
- Verify `id="main-content"` and `tabIndex={-1}` on `<main>` from Plan 02.
- Add `aria-label="Dashboard sidebar"` to the aside element.
- Ensure the overall flex container doesn't trap keyboard focus.

**Audit and fix BookingWizard.tsx:**
- The step indicator should have `role="progressbar"` or `aria-label` describing current step.
- Form inputs should have associated labels (check if react-hook-form + shadcn Form already handles this via FormLabel).
- Form error messages need `role="alert"` or `aria-live="assertive"` for screen reader announcement. Check if shadcn Form's FormMessage component already adds this. If not, update the FormMessage usage to include `role="alert"`.
- The "Next Step" and "Previous Step" buttons need descriptive labels if they only show icons.

**General patterns to verify and fix across touched files:**
1. Every `<img>` has alt text (or alt="" for decorative images)
2. Every icon-only Button has aria-label
3. Color is not the only means of conveying information (check badge/status components)
4. Focus order follows visual order (no tabIndex > 0)
5. Interactive elements have minimum 44x44px touch target (Tailwind: min-h-[44px] min-w-[44px] on small buttons)

Read each file before modifying. Make targeted, minimal changes. Do NOT rewrite entire components. Only add/modify accessibility attributes.
  </action>
  <verify>
`cd apps/web && pnpm type-check` passes. Grep for `aria-label` in sidebar.tsx (should appear at least 2 times: nav and toggle button). Grep for `aria-current` in sidebar.tsx. Grep for `role="alert"` in any modified form components. Verify no tabIndex > 0 exists in any modified file.
  </verify>
  <done>Sidebar navigation has aria-label, aria-current on active item, and aria-expanded on toggle. Header has accessible mobile menu and user menu buttons. Booking wizard forms use role="alert" for error messages. All icon-only buttons have aria-labels. Focus order is logical throughout dashboard.</done>
