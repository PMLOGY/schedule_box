<task type="auto">
  <name>Task 1: Create subscription service layer with state machine</name>
  <files>apps/web/app/api/v1/billing/service.ts</files>
  <action>
    Create `apps/web/app/api/v1/billing/service.ts` containing the subscription business logic. Follow the pattern established in `apps/web/app/api/v1/payments/service.ts` (status transitions, idempotency checks).

    **Functions to implement:**

    1. `createSubscription(companyId: number, plan: SubscriptionPlan, billingCycle: BillingCycle)`:
       - Validate plan is not 'free' (can't subscribe to free)
       - Check no existing active/trialing subscription for company (one active sub per company)
       - Insert into subscriptions table with status='trialing', period start=now, period end=now+30d (monthly)
       - Return the subscription record

    2. `transitionSubscriptionStatus(subscriptionId: number, newStatus: SubscriptionStatus, tx?: Database)`:
       - Use SELECT FOR UPDATE to lock the subscription row
       - Validate transition against VALID_SUBSCRIPTION_TRANSITIONS from shared/types/billing
       - Update status, updatedAt
       - If transitioning to 'active': update currentPeriodStart/End
       - If transitioning to 'past_due': set dunningStartedAt=now
       - If transitioning to 'expired': update companies.subscriptionPlan='free', companies.subscriptionValidUntil=null
       - Insert into subscription_events table for audit trail
       - Return updated subscription

    3. `activateSubscription(subscriptionId: number, comgateTransactionId: string)`:
       - Called by webhook on first payment success
       - Transition to 'active'
       - Store comgateInitTransactionId on subscription (this is the token for future recurring charges)
       - Update companies.subscriptionPlan to the subscription's plan
       - Update companies.subscriptionValidUntil to currentPeriodEnd

    4. `upgradeSubscription(companyId: number, newPlan: SubscriptionPlan)`:
       - Find active subscription for company
       - Validate upgrade (newPlan must be higher tier)
       - Calculate proration using calculateProration from shared/types/billing
       - If proration > 0: initiate Comgate payment with initRecurring=true for prorated amount
       - Update subscription.plan to newPlan immediately
       - Update companies.subscriptionPlan to newPlan
       - Log subscription_event with type 'plan.upgraded'
       - Return { subscription, prorationAmount, comgateRedirectUrl? }

    5. `downgradeSubscription(companyId: number, newPlan: SubscriptionPlan)`:
       - Find active subscription for company
       - Validate downgrade (newPlan must be lower tier)
       - Set subscription.cancelAtPeriodEnd with metadata about pending downgrade plan
       - Do NOT change plan immediately -- takes effect at period end
       - Log subscription_event with type 'plan.downgrade_scheduled'
       - Return updated subscription

    6. `processSubscriptionWebhook(comgateTransactionId: string, status: string, payload: Record<string, unknown>)`:
       - Check idempotency: look for existing subscription_event with this comgateTransactionId
       - Find subscription by comgateInitTransactionId or by matching a pending payment refId
       - Use SELECT FOR UPDATE on subscription row
       - If PAID: call activateSubscription or extend period (renewal)
       - If CANCELLED/FAILED: transition to past_due if active, or expired if past_due
       - Insert subscription_event for audit

    7. `getSubscriptionForCompany(companyId: number)`:
       - Query subscription with status NOT IN ('expired') for company
       - Return subscription with computed fields (daysUntilRenewal, etc.)

    8. `checkSubscriptionEventIdempotency(comgateTransactionId: string)`:
       - Check if a subscription_event with this comgateTransactionId already exists
       - Return { alreadyProcessed: boolean }

    **Important implementation notes:**
    - All state transitions MUST go through transitionSubscriptionStatus (never update status directly)
    - All webhook processing MUST use SELECT FOR UPDATE on subscription row
    - Use database transactions for all multi-table operations
    - Import PLAN_CONFIG, VALID_SUBSCRIPTION_TRANSITIONS, calculateProration, getVatRate from '@schedulebox/shared'
    - Plan tier ordering for upgrade/downgrade validation: free < essential < growth < ai_powered

  <action>
    Create 7 API route files. Follow the existing route patterns in `apps/web/app/api/v1/` (Zod validation, auth middleware, try/catch with AppError, NextRequest/NextResponse).

    **1. GET /api/v1/billing/plans** (`plans/route.ts`):
    - Public endpoint (no auth required -- visitors can see pricing)
    - Return PLAN_CONFIG from shared/types/billing as JSON array
    - Include name, price, priceAnnual, currency, features for each plan
    - Response: `{ plans: PlanInfo[] }`

    **2. POST /api/v1/billing/subscribe** (`subscribe/route.ts`):
    - Auth required (owner role)
    - Zod body: `{ plan: z.enum(['essential', 'growth', 'ai_powered']), billingCycle?: z.enum(['monthly', 'annual']) }`
    - Call createSubscription from service.ts
    - Call initComgatePayment with initRecurring=true, redirectUrl pointing to `/settings/billing?payment=pending`, callbackUrl pointing to `/api/v1/billing/webhook`
    - Store comgateTransactionId on subscription
    - Return `{ redirectUrl: string, subscriptionUuid: string }`
    - Error if company already has active subscription (409)

    **3. GET /api/v1/billing/subscription** (`subscription/route.ts`):
    - Auth required (owner role)
    - Call getSubscriptionForCompany(companyId)
    - Return current subscription with plan details, status, period dates, next renewal
    - Also include current plan's features from PLAN_CONFIG
    - Return 404 if no active subscription (company is on Free plan)

    **4. POST /api/v1/billing/upgrade** (`upgrade/route.ts`):
    - Auth required (owner role)
    - Zod body: `{ plan: z.enum(['essential', 'growth', 'ai_powered']) }`
    - Call upgradeSubscription from service.ts
    - If proration requires payment: return Comgate redirect URL
    - If no payment needed (e.g., free upgrade during trial): return updated subscription
    - Error if no active subscription (400)
    - Error if target plan is not higher tier (400)

    **5. POST /api/v1/billing/downgrade** (`downgrade/route.ts`):
    - Auth required (owner role)
    - Zod body: `{ plan: z.enum(['free', 'essential', 'growth']) }`
    - Call downgradeSubscription from service.ts
    - Return subscription with `pendingDowngrade: plan` info
    - Error if no active subscription (400)
    - Error if target plan is not lower tier (400)

    **6. GET /api/v1/billing/status** (`status/route.ts`):
    - Auth required (owner role)
    - Polling endpoint for frontend after Comgate payment redirect
    - Query latest subscription_event for company's subscription
    - Return `{ subscriptionStatus, planName, activated: boolean, lastEventType }`
    - Designed to be called every 1-2 seconds for up to 10 seconds after payment return

    **7. POST /api/v1/billing/webhook** (`webhook/route.ts`):
    - PUBLIC endpoint (no auth -- called by Comgate servers)
    - Follow the EXACT same pattern as `apps/web/app/api/v1/webhooks/comgate/route.ts`:
      a. Read raw body as text
      b. Parse as URLSearchParams
      c. Verify Comgate secret via verifyComgateWebhookSecret
      d. Extract transId and status
      e. Call processSubscriptionWebhook from service.ts
      f. Return 200 immediately (Comgate expects <5s response)
    - Defense-in-depth: cross-check status via getComgatePaymentStatus API
    - Idempotency via subscription_events table (not processed_webhooks -- these are separate)
    - On PAID: activate subscription or extend period
    - On CANCELLED: mark as failed, transition status if appropriate
    - Log all events to subscription_events

    **Auth middleware pattern:** Look at how other protected routes get the user/company context. Likely via a `getAuthContext` or similar function that extracts JWT claims. Follow the existing pattern exactly.
