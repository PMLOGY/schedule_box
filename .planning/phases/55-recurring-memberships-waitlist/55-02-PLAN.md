**apps/web/validations/recurring.ts** — Zod schemas:
- `recurringSeriesCreateSchema`: serviceId (string uuid), employeeId (string uuid, optional), customerId (string uuid), repeatPattern ('weekly' | 'biweekly' | 'monthly'), startDate (string ISO date), endDate (string ISO date, optional), startTime (string HH:mm), durationMinutes (number 5-480), maxOccurrences (number 1-52, optional), notes (string, optional). Refine: either endDate or maxOccurrences must be provided (not both null). Default maxOccurrences to 12 if neither provided.
- `recurringSeriesUpdateSchema`: partial of create schema (all optional)
- `occurrenceEditSchema`: startTime (datetime string, optional), endTime (datetime string, optional), employeeId (string uuid, optional), notes (string, optional)

**apps/web/lib/booking/recurring-service.ts** — Service functions:

`createRecurringSeries(data, companyId)`:
1. Resolve UUIDs to internal IDs (service, employee, customer) using existing patterns from booking-service.ts
2. Insert into recurringSeries table
3. Generate occurrence dates based on repeatPattern: weekly (+7 days), biweekly (+14 days), monthly (same day next month). Respect endDate and maxOccurrences limits. Cap at 52 occurrences max for safety.
4. For each occurrence date, combine with startTime to get startTime/endTime timestamps. Check employee availability (reuse the same availability check from createBooking — look at how booking-service.ts checks for double-booking with SELECT FOR UPDATE). Skip occurrences that conflict — log warning but don't fail the whole series.
5. Call createBooking for each valid occurrence, passing recurringSeriesId. Each occurrence gets its own booking row with recurringSeriesId set.
6. Return the series with count of created/skipped occurrences.

`editSingleOccurrence(bookingId, data, companyId)`:
1. Find the booking, verify it has recurringSeriesId (error if not a recurring occurrence)
2. Update the individual booking using updateBooking from booking-service.ts
3. Does NOT touch other occurrences or the series record

`editSeries(seriesId, data, companyId)`:
1. Find the series record
2. Update the series record itself (pattern, times, etc.)
3. Find all future occurrences (startTime > now) with this recurringSeriesId
4. Update each future occurrence — recalculate times based on new pattern/startTime. If employee changed, re-check availability. Skip conflicts.
5. Return updated series with count of modified occurrences

`cancelOccurrence(bookingId, companyId, cancelledBy)`:
1. Find booking, verify recurringSeriesId
2. Use booking-transitions.ts cancelBooking to cancel this single occurrence
3. Does NOT affect other occurrences

`cancelSeries(seriesId, companyId, cancelledBy)`:
1. Set series isActive = false
2. Find all future non-cancelled occurrences with this recurringSeriesId
3. Cancel each using booking-transitions.ts cancelBooking
4. Return count of cancelled occurrences

`getRecurringSeries(seriesId, companyId)` — fetch series with occurrence count
`listRecurringSeries(companyId, query)` — list with pagination

All functions must use tenant isolation (companyId in every query). Use dbTx for multi-row operations.

**apps/web/app/api/v1/bookings/recurring/route.ts**:
- POST: requiresAuth, BOOKINGS_CREATE permission. Validate body with recurringSeriesCreateSchema. Call createRecurringSeries. Return 201 with series data + occurrence count.
- GET: requiresAuth, BOOKINGS_READ permission. Call listRecurringSeries with pagination query. Return paginated response.

**apps/web/app/api/v1/bookings/recurring/[id]/route.ts**:
- GET: requiresAuth, BOOKINGS_READ. Parse id param (UUID). Call getRecurringSeries. Return series with occurrences.
- PUT: requiresAuth, BOOKINGS_UPDATE. Validate body with recurringSeriesUpdateSchema. Call editSeries. Return updated series.
- DELETE: requiresAuth, BOOKINGS_DELETE. Call cancelSeries. Return 200 with cancellation count.

**apps/web/app/api/v1/bookings/recurring/[id]/occurrences/route.ts**:
- GET: List all occurrences (bookings) for this series. Query bookings WHERE recurringSeriesId = series internal id. Return as paginated booking list.
- PUT: Edit a single occurrence. Body includes bookingId + occurrence edit data. Call editSingleOccurrence.
- DELETE: Cancel a single occurrence. Body includes bookingId. Call cancelOccurrence.

Check if PERMISSIONS object already has BOOKINGS_CREATE/READ/UPDATE/DELETE — reuse those (recurring bookings are just bookings with a series link). If not, add to the RBAC permissions file.
