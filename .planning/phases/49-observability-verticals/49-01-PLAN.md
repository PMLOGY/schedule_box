1. Install packages in apps/web:
   `pnpm --filter @schedulebox/web add @vercel/otel @opentelemetry/sdk-logs @opentelemetry/api-logs @opentelemetry/instrumentation`

2. Update `apps/web/instrumentation.ts`:
   - Import `registerOTel` from `@vercel/otel`
   - Call `registerOTel({ serviceName: 'schedulebox' })` at the TOP of `register()` function, BEFORE the NEXT_RUNTIME guards — no runtime guard around it (per research pitfall #1). @vercel/otel handles both nodejs and edge internally.
   - Keep existing Sentry imports and runtime-guarded Sentry config imports exactly as they are.

3. Update `apps/web/middleware.ts`:
   - Generate `const requestId = crypto.randomUUID()` at the start of the middleware function.
   - After `intlMiddleware(req)` call returns, set `response.headers.set('X-Request-Id', requestId)` on the response.
   - Also set `response.headers.set('x-request-id', requestId)` for downstream API route consumption.
   - Return the modified response. If intlMiddleware returns a redirect or next, attach X-Request-Id to whichever response object it produces.

4. Create `apps/web/lib/logger/route-logger.ts`:
   - Import `logInfo` and `logError` from `@schedulebox/shared`.
   - Export `logRouteComplete(params: { route: string; method: string; status: number; duration_ms: number; request_id: string; error?: Error })` function.
   - If params.error exists, call `logError('route_error', { route, method, status, duration_ms, request_id, error_message: error.message })`.
   - Otherwise call `logInfo('route_complete', { route, method, status, duration_ms, request_id })`.
   - Export a `getRequestId(req: Request): string` helper that reads `req.headers.get('x-request-id') ?? 'unknown'`.

IMPORTANT: Do NOT import `initTracer` from `packages/shared/src/telemetry/tracer.ts` anywhere in apps/web. The old OTLP gRPC tracer conflicts with @vercel/otel (pitfall #2).

  <action>
1. For each of the 3 critical API route files listed above:
   - Import `trace` from `@opentelemetry/api`.
   - Import `logRouteComplete` and `getRequestId` from `@/lib/logger/route-logger`.
   - At the start of each handler (GET, POST, PUT, etc.), capture `const startTime = Date.now()` and `const requestId = getRequestId(req)`.
   - Wrap the core logic in `trace.getTracer('schedulebox').startActiveSpan('schedulebox.{route_name}.{method}', async (span) => { ... })`.
   - Set relevant span attributes: `span.setAttributes({ 'http.route': '/api/v1/...', 'booking.company_id': '...' })` where applicable.
   - In the finally block: `span.end()`.
   - After the response is built, call `logRouteComplete({ route: '/api/v1/...', method: 'POST', status: 200, duration_ms: Date.now() - startTime, request_id: requestId })`.
   - In catch blocks: call `span.recordException(err)` and `logRouteComplete({ ..., status: 500, error: err })`.

2. Route-specific spans:
   - `/api/v1/public/company/[slug]/bookings POST`: span name `schedulebox.booking.create`, attributes: company_slug, service_id.
   - `/api/v1/bookings GET`: span name `schedulebox.booking.list`, attributes: company_id.
   - `/api/v1/auth/login POST`: span name `schedulebox.auth.login`, attributes: (no PII — just success/failure status).

3. Keep all existing route logic exactly as-is. Only wrap with spans and add logging calls. Do not change any business logic.
