  <name>Task 1: Double-booking prevention integration tests</name>
  <files>tests/integration/booking/double-booking.test.ts</files>
  <action>
    Create `tests/integration/booking/double-booking.test.ts` testing ITEST-02 requirements.

    **Setup pattern:**
    - `import { inject } from 'vitest'`
    - Import `postgres` and `drizzle` directly (not through test-db helper, because concurrent tests need independent connections with higher max pool)
    - Import schema `* as schema from '@schedulebox/database'`
    - Import seed helpers: `seedCompany, seedService, seedEmployee, seedCustomer, seedEmployeeService`
    - Import `truncateAllTables` from helpers/test-db
    - In `beforeAll`: create superuser client with `postgres(inject('DATABASE_URL'), { max: 10 })` (need 10 connections for concurrent transaction tests)
    - In `beforeEach`: truncate all tables, seed fresh data: one company, one service (60 min), one employee, one customer, link employee to service via seedEmployeeService
    - In `afterAll`: close client

    **Test cases:**

    1. `'concurrent bookings to same slot - exactly one succeeds'`:
       - Define a timeslot: startTime = tomorrow 10:00, endTime = tomorrow 11:00
       - Create TWO independent postgres clients from the same DATABASE_URL (each with max: 2)
       - Use `Promise.allSettled` with two concurrent transactions:
         - Transaction A: SELECT FOR UPDATE on employee, check for conflicts, INSERT booking
         - Transaction B: Same sequence, same employee + timeslot
       - Assert: exactly 1 fulfilled, exactly 1 rejected (or the second finds a conflict and throws)
       - Close both independent clients after test
       - NOTE: The SELECT FOR UPDATE pattern means the second transaction blocks until the first commits, then sees the conflict. The key assertion is that only 1 booking row exists for that slot.

    2. `'btree_gist exclusion constraint rejects overlapping INSERT'`:
       - Insert a confirmed booking for employee at 10:00-11:00 (via seed helper)
       - Attempt a direct INSERT of another booking for same employee at 10:30-11:30 with status 'pending'
       - Assert: INSERT throws an error containing 'no_overlapping_bookings' or 'conflicting key value' or 'exclusion constraint'
       - This tests the DATABASE-LEVEL safety net independent of application logic

    3. `'non-overlapping bookings for same employee succeed'`:
       - Insert booking A for 10:00-11:00
       - Insert booking B for 11:00-12:00 (adjacent, not overlapping)
       - Assert: both bookings exist in DB (2 rows for this employee)

    4. `'cancelled booking does not block new booking for same slot'`:
       - Insert booking with status 'cancelled' for 10:00-11:00
       - Insert new booking for same employee at 10:00-11:00 with status 'pending'
       - Assert: new booking succeeds (exclusion constraint WHERE clause excludes cancelled)

    **Important implementation notes:**
    - Use raw `client.begin(async (tx) => { ... })` for concurrent transaction tests, NOT Drizzle's `db.transaction()`. The postgres.js `begin()` gives direct control over transaction isolation.
    - For the concurrent test, create separate postgres() clients so they get truly independent connections (same-client connections may be serialized by the pool).
    - When inserting bookings, provide ALL required non-null fields: companyId, customerId, serviceId, employeeId, startTime, endTime, status, price, currency, source.
    - Use `eq` and `and` from `drizzle-orm` for queries.
  </action>
  <verify>
    Run `pnpm test:integration` -- all 4 double-booking tests pass. Specifically verify:
    - Concurrent test shows exactly 1 success + 1 failure
    - Exclusion constraint test shows rejection error
    - Non-overlapping test shows 2 bookings
    - Cancelled slot test shows new booking succeeds
  </verify>
  <done>
    Double-booking prevention validated at both application level (concurrent SELECT FOR UPDATE) and database level (btree_gist exclusion constraint). Cancelled bookings correctly excluded from constraint.
  </done>

<task type="auto">
  <name>Task 2: RLS multi-tenant isolation integration tests</name>
  <files>tests/integration/rls/tenant-isolation.test.ts</files>
  <action>
    Create `tests/integration/rls/tenant-isolation.test.ts` testing ITEST-03 requirements.

    **Setup pattern:**
    - Import `inject` from vitest
    - Import `postgres` and `drizzle`
    - Import seed helpers for companies, users, services, employees, customers, bookings
    - Import `truncateAllTables` from helpers/test-db
    - In `beforeAll`:
      - Create superuser client from `inject('DATABASE_URL')` for seeding (bypasses RLS)
      - Create app client from `inject('DATABASE_URL_APP')` for RLS-enforced queries
      - Seed TWO companies (Company A and Company B) with superuser
      - Seed data for EACH company: a user, a service, an employee, a customer, a booking
      - Store company IDs, user IDs for RLS context
    - In `afterAll`: close both clients

    **CRITICAL:** All RLS-enforced queries MUST use the `appClient` (non-superuser). Data seeding uses `superClient` to bypass RLS. Using superuser for RLS queries would make all tests pass falsely.

    **Test cases for customers table:**

    1. `'Company A sees only its own customers'`:
       - Use appClient.begin() transaction
       - SET LOCAL app.company_id = companyAId, app.user_role = 'owner'
       - SELECT * FROM customers
       - Assert: all returned rows have company_id = companyAId
       - Assert: count matches expected (1 customer seeded for company A)

    2. `'Company B sees only its own customers'`:
       - Same pattern with companyBId
       - Assert: all rows have company_id = companyBId

    3. `'Company A cannot see Company B customers even with explicit WHERE'`:
       - SET LOCAL context to company A
       - SELECT * FROM customers WHERE company_id = companyBId (explicitly trying to access B's data)
       - Assert: returns 0 rows (RLS filters BEFORE WHERE clause)

    **Test cases for bookings table:**

    4. `'Company A sees only its own bookings'`:
       - SET LOCAL context to company A
       - SELECT * FROM bookings
       - Assert: all rows have company_id = companyAId

    5. `'Company B sees only its own bookings'`:
       - Same with company B

    **Test cases for services table:**

    6. `'Company A sees only its own services'`:
       - SET LOCAL context to company A
       - SELECT * FROM services
       - Assert: all rows have company_id = companyAId

    **Test cases for employees table:**

    7. `'Company A sees only its own employees'`:
       - SET LOCAL context to company A
       - SELECT * FROM employees
       - Assert: all rows have company_id = companyAId

    **Cross-table isolation test:**

    8. `'RLS prevents cross-tenant data access across all key tables'`:
       - SET LOCAL context to company A
       - Query customers, services, employees, bookings in one transaction
       - Assert: all results are scoped to company A
       - Do same for company B
       - Assert: result sets are completely disjoint (no shared IDs)

    **INSERT isolation test:**

    9. `'RLS prevents inserting data for another company'`:
       - SET LOCAL context to company A
       - Attempt INSERT into customers with company_id = companyBId
       - Assert: INSERT fails or the row is not visible to company B (depending on RLS policy type -- if USING only, insert may succeed but SELECT won't show it; if WITH CHECK, insert fails)
       - NOTE: Check the actual RLS policy type from policies.sql. If policies use USING without WITH CHECK, the insert may succeed but be invisible. If WITH CHECK exists, the insert will be rejected. Document whichever behavior occurs.

    **Implementation notes:**
    - Use `appClient.begin(async (tx) => { ... })` for all RLS queries
    - Use `SET LOCAL` (not `SET`) so the session variable is scoped to the transaction and doesn't leak
    - Query via raw SQL `tx\`SELECT * FROM customers\`` rather than Drizzle ORM to directly test PostgreSQL RLS behavior
    - Seed employees with linked employee_services and unique emails per company to avoid unique constraint conflicts
  </action>
  <verify>
    Run `pnpm test:integration` -- all RLS tests pass. Specifically verify:
    - Company A sees exactly 1 customer (its own)
    - Company B sees exactly 1 customer (its own)
    - Cross-company explicit WHERE returns 0 rows
    - Same isolation holds for bookings, services, employees
  </verify>
  <done>
    Multi-tenant RLS isolation validated: two companies cannot access each other's data in customers, services, employees, and bookings tables. Non-superuser role correctly enforces RLS policies.
  </done>
